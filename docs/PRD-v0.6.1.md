## 2. 需求清单

### R1 · Markdown 导出保留表格结构(2026-07-20 安哥)

**原始需求**:下载含表格的文章,存成 markdown 时表格格式被直接忽略,只剩表格里的文本,能否优化。

**现状核实**(2026-07-20 复现):属实。`export-markdown.ts` 的 turndown 实例只配了 `headingStyle`/`codeBlockStyle` + 一条微信代码块规则,**没有表格规则**;turndown 内核不含表格支持(GFM 表格是扩展语法),遇到 `<table>` 会退化为逐单元格取文本。实测:

```
输入 <table><thead><tr><th>列A</th><th>列B</th></tr></thead>
     <tbody><tr><td>甲</td><td>1</td></tr><tr><td>乙</td><td>2</td></tr></tbody></table>
当前输出 "列A\n\n列B\n\n甲\n\n1\n\n乙\n\n2"   ← 结构全丢,行列关系不可读
期望输出 | 列A | 列B |
        | --- | --- |
        | 甲 | 1 |
        | 乙 | 2 |
```

**影响面**:只影响 md 格式。html/pdf 走原始 HTML 渲染,表格本来就正常;GUI 阅读器的 md 视图用 `react-markdown` + `remark-gfm`(已装),**一旦 md 里有正确的 GFM 表格就能直接渲染**,无需改阅读器。

**细化方案**:

- 给 turndown 增加 GFM 表格规则。两条路线,倾向前者:
  1. **引入 `turndown-plugin-gfm`**(turndown 官方生态插件,含 table/strikethrough/taskListItems):成熟、省事;需新增一个依赖(体积小、无原生依赖)。
  2. 自写 table 规则(约 30–50 行):不加依赖,但要自行处理表头缺失、跨行跨列、单元格内换行等边界——重复造轮子,除非依赖审计有顾虑,否则不划算。
- **微信表格的真实结构需先抓样本核实**(公众号编辑器可能产出 `<table>` 无 `<thead>`、或用 `<section>` 模拟表格):按样本决定是否需要「无表头时补一行空表头」等兜底(GFM 表格语法要求必须有分隔行)。
- **边界处理**(实现时逐条对样本验证):无 `<thead>` → 用首行 `<tr>` 当表头;单元格含换行/`<br>` → 转成 `<br>` 或空格(GFM 表格不允许裸换行);单元格含管道符 `|` → 转义;跨行跨列(rowspan/colspan)→ 尽力而为(GFM 不支持合并单元格,按拆分呈现并在 devlog 记录取舍)。
- **不改动**:html/pdf/meta 导出、阅读器渲染链路。

**验收(草)**:
- [ ] 标准表格(带 thead)转出合法 GFM 表格,GUI 阅读器 md 视图正常渲染为表格。
- [ ] 无 thead 的表格自动以首行为表头,不产出非法 markdown。
- [ ] 单元格内含 `|`、换行、行内格式(粗体/链接)不破坏表格结构。
- [ ] 真实微信含表格文章端到端验证(下载 → content.md 结构正确 → 阅读器渲染)。
- [ ] 既有 md 导出测试全部保持通过(代码块、图片、标题等不受影响)。

### R2 · 应用内可见版本号(2026-07-20 安哥)

**原始需求**:打开的窗口中不知晓当前使用的版本号。

**现状核实**(2026-07-20):版本号在 GUI 里**完全不可见**——只有 CLI `--version` 能拿到。`electron/preload.ts` / `src/renderer/api.ts` 均无 version 通道,设置页也没有「关于」区块。用户升级后无法确认装的是哪一版(安哥手上就长期停在 v0.5.2 而不自知)。

**细化方案**:

- 新增最小 IPC:`app:version`(主进程 `app.getVersion()`)→ preload → `api.appVersion()`。
- **呈现位置(倾向 A,待安哥定)**:
  - **A. 刊头品牌旁**:`MainLayout` 的 `wx-kit` 标记后缀一个淡色 `v0.6.1`——常驻可见、零点击成本,占位极小(与「不要让用户思考」一致)。
  - **B. 设置页「关于」区块**:版本号 + 仓库/Release 链接 + 「检查更新」的未来位;信息更全但要点进去才看到。
  - 两者可同时做:刊头常驻数字,设置页放完整信息(推荐组合)。
- 版本号取自 `app.getVersion()`(读 package.json version),**不硬编码**,避免发版漏改。

**验收(草)**:
- [ ] GUI 内可见当前版本号,与 `package.json`/CLI `--version` 一致。
- [ ] 打包态显示正确(非 dev 的 0.0.0-dev 之类)。

### R3 · 窗口标题栏的「wx-kit」是否有必要(2026-07-20 安哥)

**原始需求**:窗口最顶部标题栏(与最小化/最大化按钮同一行)写着「wx-kit」,按现代 UI 审美有必要吗?

**现状核实**:`electron/main.ts:44` 创建窗口时设 `title: 'wx-kit'`,用的是系统默认标题栏(未设 `titleBarStyle`),macOS 上表现为一条 28px 高的灰色条,居中一个「wx-kit」——而应用内刊头已经有「微信百宝箱 WX-KIT」品牌区,**信息重复**。

**评估**:安哥的判断成立。现代 mac 应用(VS Code、Notion、Linear、Figma)普遍做法是隐藏原生标题栏、把红绿灯按钮嵌入自绘顶栏,省下一条高度、视觉更整。可选路线:

| 方案 | 做法 | 代价 |
|---|---|---|
| **A. `titleBarStyle: 'hiddenInset'`(倾向)** | 隐藏标题栏,红绿灯浮在内容区左上;刊头左侧留 ~78px 空位避让 | 需调 `.masthead` 左内边距;win/Linux 需分平台处理(win 无红绿灯,或保留默认标题栏) |
| B. 保留标题栏但改文案 | 如显示「微信百宝箱」或当前页名 | 仍占一条高度,治标 |
| C. 完全无边框 `frame: false` | 自绘全部窗口控件 | 要自实现拖拽/最小化/最大化/关闭,成本与风险最高,不建议 |

**待安哥定**:选 A(隐藏标题栏、刊头避让红绿灯)还是 B(仅改文案)?若选 A,Windows 侧策略也需一并定(建议 win 维持默认标题栏,只 mac 隐藏——win 的窗口按钮在右上,自绘避让规则不同)。

**验收(草)**:
- [ ] mac 上窗口顶部不再有重复的「wx-kit」条,红绿灯与刊头不重叠、不遮挡。
- [ ] 拖动窗口、双击最大化等原生行为保持可用。
- [ ] e2e 与真机截图确认布局无错位(含最大化/还原)。
