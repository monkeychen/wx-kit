## 2. 需求清单

### R1 · Markdown 导出保留表格结构(2026-07-20 安哥)

**原始需求**:下载含表格的文章,存成 markdown 时表格格式被直接忽略,只剩表格里的文本,能否优化。

**现状核实**(2026-07-20 复现):属实。`export-markdown.ts` 的 turndown 实例只配了 `headingStyle`/`codeBlockStyle` + 一条微信代码块规则,**没有表格规则**;turndown 内核不含表格支持(GFM 表格是扩展语法),遇到 `<table>` 会退化为逐单元格取文本。实测:

```
输入 <table><thead><tr><th>列A</th><th>列B</th></tr></thead>
     <tbody><tr><td>甲</td><td>1</td></tr><tr><td>乙</td><td>2</td></tr></tbody></table>
当前输出 "列A\n\n列B\n\n甲\n\n1\n\n乙\n\n2"   ← 结构全丢,行列关系不可读
期望输出 | 列A | 列B |
        | --- | --- |
        | 甲 | 1 |
        | 乙 | 2 |
```

**影响面**:只影响 md 格式。html/pdf 走原始 HTML 渲染,表格本来就正常;GUI 阅读器的 md 视图用 `react-markdown` + `remark-gfm`(已装),**一旦 md 里有正确的 GFM 表格就能直接渲染**,无需改阅读器。

**细化方案**:

- 给 turndown 增加 GFM 表格规则。两条路线,倾向前者:
  1. **引入 `turndown-plugin-gfm`**(turndown 官方生态插件,含 table/strikethrough/taskListItems):成熟、省事;需新增一个依赖(体积小、无原生依赖)。
  2. 自写 table 规则(约 30–50 行):不加依赖,但要自行处理表头缺失、跨行跨列、单元格内换行等边界——重复造轮子,除非依赖审计有顾虑,否则不划算。
- **微信表格的真实结构需先抓样本核实**(公众号编辑器可能产出 `<table>` 无 `<thead>`、或用 `<section>` 模拟表格):按样本决定是否需要「无表头时补一行空表头」等兜底(GFM 表格语法要求必须有分隔行)。
- **边界处理**(实现时逐条对样本验证):无 `<thead>` → 用首行 `<tr>` 当表头;单元格含换行/`<br>` → 转成 `<br>` 或空格(GFM 表格不允许裸换行);单元格含管道符 `|` → 转义;跨行跨列(rowspan/colspan)→ 尽力而为(GFM 不支持合并单元格,按拆分呈现并在 devlog 记录取舍)。
- **不改动**:html/pdf/meta 导出、阅读器渲染链路。

**验收(草)**:
- [ ] 标准表格(带 thead)转出合法 GFM 表格,GUI 阅读器 md 视图正常渲染为表格。
- [ ] 无 thead 的表格自动以首行为表头,不产出非法 markdown。
- [ ] 单元格内含 `|`、换行、行内格式(粗体/链接)不破坏表格结构。
- [ ] 真实微信含表格文章端到端验证(下载 → content.md 结构正确 → 阅读器渲染)。
- [ ] 既有 md 导出测试全部保持通过(代码块、图片、标题等不受影响)。

### R2 · 应用内可见版本号(2026-07-20 安哥)

**原始需求**:打开的窗口中不知晓当前使用的版本号。

**现状核实**(2026-07-20):版本号在 GUI 里**完全不可见**——只有 CLI `--version` 能拿到。`electron/preload.ts` / `src/renderer/api.ts` 均无 version 通道,设置页也没有「关于」区块。用户升级后无法确认装的是哪一版(安哥手上就长期停在 v0.5.2 而不自知)。

**细化方案**:

- 新增最小 IPC:`app:version`(主进程 `app.getVersion()`)→ preload → `api.appVersion()`。
- **呈现位置(2026-07-20 安哥定:两处都做)**:
  - **刊头品牌旁**:`MainLayout` 的 `WX-KIT` 标记后缀一个淡色小号 `v0.7.0`——常驻可见、零点击成本,占位极小;样式须克制(字号/色阶低于品牌标记,不喧宾夺主)。
  - **设置页「关于」区块**:版本号 + 项目主页/Releases 链接(`shell.openExternal`),作为将来「检查更新」的落位;信息完整、可复制。
- 版本号取自 `app.getVersion()`(读 package.json version),**不硬编码**,避免发版漏改。

**验收(草)**:
- [ ] 刊头常驻显示版本号,与 `package.json`/CLI `--version` 一致;样式不喧宾夺主(真机截图确认)。
- [ ] 设置页「关于」区块显示版本号 + 可点开的项目主页/Releases 链接。
- [ ] 打包态显示正确(不是 dev 的 `0.0.0-dev`)。

### R3 · 窗口标题栏的「wx-kit」是否有必要(2026-07-20 安哥)

**原始需求**:窗口最顶部标题栏(与最小化/最大化按钮同一行)写着「wx-kit」,按现代 UI 审美有必要吗?

**现状核实**:`electron/main.ts:44` 创建窗口时设 `title: 'wx-kit'`,用的是系统默认标题栏(未设 `titleBarStyle`),macOS 上表现为一条 28px 高的灰色条,居中一个「wx-kit」——而应用内刊头已经有「微信百宝箱 WX-KIT」品牌区,**信息重复**。

**评估与选型过程**(2026-07-20 与安哥当场论证):

先考虑过隐藏原生标题栏(`titleBarStyle: 'hiddenInset'`,红绿灯嵌入自绘顶栏,VS Code/Linear 那种形态)。实测量化:当前标题栏 **32px**、内容区 768/800;隐藏后标题栏 0、内容区占满。但安哥指出**刊头本身高 68px**(`.masthead`),红绿灯(直径 ~12px)嵌进去会违和——垂直居中则按钮悬在半空、与 mac「窗口左上角」的肌肉记忆不符,贴顶对齐则刊头下半空出、品牌与按钮不同线。**要做好必须连刊头一起瘦身到 44–48px,那是顶栏重新设计,会动到这套 UI 的「书刊感」气质,不属于本次问题范围。**

**定案(安哥选 3):保留原生标题栏,只清掉重复文案**——`BrowserWindow` 的 `title` 设为空串,mac 上标题栏只剩红绿灯 + 一条干净浅灰条,刊头原样不动。一行改动、零风险,精确解决「wx-kit 出现两次」的信息重复。

- 实现:**两处都要改,只改一处无效**——① `electron/main.ts` 创建窗口的 `title: 'wx-kit'`;② **`index.html` 的 `<title>wx-kit</title>`**(已核实存在):页面加载后 HTML title 会覆盖 BrowserWindow 的 title,只改 main.ts 会被它顶回来。可给 BrowserWindow 加 `titleBarStyle` 无关,直接把 `<title>` 清空或去掉即可(dist/index.html 是构建产物,改源码 `index.html` 后重新 build 生效)。
- 记录取舍:「隐藏标题栏 + 刊头瘦身」作为将来 UI 改版的候选,不在本版做。

**验收(草)**:
- [ ] mac 窗口标题栏不再显示「wx-kit」文字,红绿灯与拖拽/双击最大化等原生行为不变。
- [ ] 刊头布局零变化(与改动前截图逐像素对比无差异)。
- [ ] 打包态确认(dev 与打包路径的 HTML title 都不会把文字顶回来)。

### R4 · 打通「导出素材 → agent 创作」的断点(2026-07-20 安哥)

**原始需求**:文库选文章 → 点「导出为素材」后,用户要离开窗口、找到目标文件夹、复制 json 路径、再打开 Claude Code 等 agent 工具才能开始创作,工作流很不顺畅。

**现状核实**(2026-07-20):`Library.tsx` 的 `exportMaterial` 导出后只弹一条 message(点击可 `reveal` 打开文件夹),路径要用户自己看、自己记、自己粘。断点有三处:**找路径**、**切工具**、**把上下文交给 agent**。

**方案分层(由轻到重,建议 a+b 先落地,c 单独评估)**:

- **a. 一键复制「给 agent 的指令」**(最小可用,强烈推荐先做):导出成功后,除「打开文件夹」外提供**「复制指令」**按钮,复制到剪贴板的是一段可直接粘给任意 agent 的自然语言 + 路径,例如:
  ```
  用 wx-kit 导出的素材写作:素材清单在 /abs/path/material-xxx.json,
  每篇的正文在其 contentPath 字段指向的 content.md。请先读清单再与我确认选题。
  ```
  用户只需 ⌘V 到 Claude Code——**省掉找路径、拼提示词两步**,零新依赖。
- **b. 导出结果就地可见可用**:导出后在结果提示/文库页内展示清单路径本身(可点复制路径、可点打开文件夹),不必去 Finder 里找。
- **c. 直接唤起 agent(需调研,可能不做)**:探索一键把上下文送进 Claude Code 的可行性,备选:
  - 生成一条 `claude` CLI 命令(带提示词)一并复制/执行;
  - 走 `shell.openExternal` 的 URL scheme(需确认 Claude Code 是否提供);
  - 落一个 `.md` 提示词文件到素材目录,agent 侧读它。
  **实现前必须先核实 Claude Code 的实际集成能力**,不确定就退回 a+b(不为想象中的接口写代码)。

**边界**:wx-kit 只负责「把素材和上下文准备到位、递到手边」,**不内置创作**(延续 v0.4.0 决策:wx-kit 只供料);真正的写作仍由 `agent/wx-kit-compose` 等 skill 在 agent 侧完成。

**验收(草)**:
- [ ] 导出后可一键复制「可直接粘给 agent 的指令」,内容含清单绝对路径与用法说明。
- [ ] 导出结果在应用内即可看到路径/复制路径/打开文件夹,不必去 Finder 翻找。
- [ ] 真机演练:文库选 3 篇 → 导出 → 复制 → 粘进 Claude Code → agent 能据此读到素材,全程不手动输路径。
- [ ] (若做 c)Claude Code 集成路径经真实验证可用;不可用则如实记录并只保留 a+b。

## 4. 非目标

- **项目更名(2026-07-20 提出后当场撤回)**——安哥原计划从「微信工具箱」改为面向多平台的中性名(候选 `inkbox`/墨匣、`sourcebox`、`harvestbox`、`zhaiji`,npm 主包名均实查可用;`clipkit` 已被占用)。撤回理由(安哥):**名字的具体性是获客资产**——「微信」搜得到、一眼懂、精准命中需求人群,用户下载后发现还有更多功能反而是惊喜;抽象名冷启动吃亏。改名成本亦高:brew tap 仓库名与 cask、npm 包名、CLI 命令名(用户肌肉记忆 + skill 引用)、bundle id(**改则老用户设置/session 丢失,需迁移逻辑**)、文档全量。**结论:维持 wx-kit;待多平台功能真正落地、有用户基数时再议改名**(届时改名有人跟随,且可借势宣传)。
