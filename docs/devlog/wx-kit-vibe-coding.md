# wx-kit Vibe-Coding 全程复盘

> 起始：2026-06-06 · 持续更新（活文档，每完成一个里程碑增补）　|　记录人：安哥 × Claude Code（Opus 4.8）
> 用途：个人复盘 / 方法论总结 / 对外分享的原始素材
> 范围：从把 `x-downloader` 技术探索原型产品化为 `wx-kit` 起，逐里程碑记录全程。当前已覆盖 M1（核心层 + CLI 下载）、M2（GUI），均合入 main。

## 0. TL;DR

一次会话里，从"分析一个原型"开始，走完了 **分析 → 头脑风暴对齐 → PRD → 实现计划 → 分子代理执行 → 两阶段评审 → 端到端验证 → 合并 → 跨会话续接文档** 的完整产品工程闭环，交付两个里程碑。

**成果数据**
- main 上 51 次提交，干净的语义化历史（feat / fix / docs / test / chore）。
- 62 个 vitest 单测 + 1 个 Playwright Electron 端到端测试，全绿。
- 两个里程碑（M1 核心层+CLI、M2 GUI）各自经 feature 分支开发、评审、合并。
- 产出文档：PRD、M1/M2 实现计划、AGENTS.md（含状态/路线图/陷阱）、本复盘。
- **评审与 e2e 共拦下 ~20 个真实缺陷**，其中 3 个是单测覆盖不到、只有端到端才暴露的硬坑。

**一句话心得**：Vibe-coding 不是"放飞地让 AI 写"，而是**用纪律化的流程把"感觉"约束成可验证的工程**——先定约束，再分步执行，每步都有 gate。

## 1. 全程时间线

### 阶段 1：分析原型，讲真话
用户：「分析一下该项目」。
我读完 `x-downloader`（Electron + React + Python 边车爬虫 + AnyProxy 代理 + 离线授权）后，没有只夸，而是直指问题：命名与产品不符、**三套抓取机制冗余**（代理捕获 / 隐藏窗口搜索 / Python Playwright 后台爬取）、授权只签 machineId 形同"防君子"、Python 服务默认绑 `0.0.0.0`、代理崩溃后系统代理不还原会让用户断网。判断这是 PyQt→Electron 迁移的历史包袱。

> 心法：**第一性原理 + 不谄媚**。分析的价值在于指出问题与本质，而非复述代码。

### 阶段 2：产品化需求 + 约束先行
用户明确：这是原型，要产品化为 `wx-kit`（微信百宝箱，第一阶段=下载器），需求=URL 多格式下载 / 公众号批量 / 浏览 md·html / 弃用代理，新仓库位置给定，**先出 PRD 确认再下一步**。

> 心法：**约束先行**。不直接写代码，先立"要做什么、边界在哪"。

### 阶段 3：头脑风暴，把模糊需求逼成明确决策
用 `brainstorming` skill，一次一个问题（多选），敲定了几个会重塑架构的分叉：
1. 代理弃用后批量爬取入口 → **扫码登录公众号后台 + 调图文接口**（唯一可行的非代理路径）。
2. 爬取技术形态 → **纯 Node/Electron 重写，去掉 Python 边车**（去掉 PyInstaller/Playwright/随机端口/0.0.0.0 一整套包袱）。
3. 浏览形态 → **应用内置阅读器**。
4. 授权系统 → **第一阶段不做**。
随后用户追加两条需求（多 URL 输入、面向 AI agent 的 CLI），又敲定：
5. CLI 运行形态 → **统一 Electron 二进制双模式（GUI/CLI）**，让 PDF（printToPDF）与扫码登录都能复用 Electron，零额外重型依赖。

> 心法：**为决策设计提问**。每个问题的答案都改变后续架构，所以一次一个、带推荐与理由。

### 阶段 4：PRD → 迭代 → 收敛开放问题
写出 PRD（定位/目标与非目标/F1–F5/格式规格/存储结构/架构/风控/里程碑/验收），用户审阅后追加（文章管理删除+按文件名搜索、下载进度信息），再把两个开放问题敲定（search 返回候选列表、文章库目录可配置且默认用户文档目录）。

### 阶段 5：实现计划（M1）
用 `writing-plans` skill。因第一阶段体量大，**按 PRD 的 M1–M4 里程碑拆成独立计划，先写 M1**（17 个 bite-sized 任务，含完整代码、TDD 步骤、运行命令、自检）。

### 阶段 6：分子代理执行 M1
用 `subagent-driven-development`：
- 先 `using-git-worktrees` → 用户选**就地开 feature 分支**（greenfield 不必 worktree）。
- 17 个计划任务归并成 ~8 个执行单元；**每单元：派一个全新 implementer 子代理 → spec 合规评审 → 代码质量评审 → 据评审修复 → 复验**。
- 纯逻辑严格 TDD；网络/Electron 部分注入依赖 + 手动验证。
- **端到端验证我亲自跑**（子代理沙箱解析不了 electron 二进制）：真实下载产出含合法 PDF、本地化图片、库索引；去重生效。
- `finishing-a-development-branch` → 合并 main，删分支。

### 阶段 7：M2（GUI）同流程
先问 UI 栈（选 **Antd 6 + Tailwind**）→ 写 M2 计划（10 任务）→ 分子代理执行：应用壳、三个可测接缝（Settings/wxfile 协议/读文章）、IPC 桥、下载页、文章库页、阅读器、设置页。每单元两阶段评审。我做 electron 启动冒烟 + 自动化验证。合并 main。

### 阶段 8：把"手动 GUI 验证"也自动化
用户追问「不能自动化测试吗」。我用 **Playwright 的 `_electron` 驱动器**写端到端：启动真实应用、对内置 fixture 服务跑「下载→文章库→阅读器→删除→设置」，**断言阅读器图片 `naturalWidth>0`** 来证明 `wxfile://` 协议确实把本地图片喂进了页面。过程中抓到一个**测试选择器**的坑（Antd v6 会在两汉字间插空格，"阅 读"），修正后全绿，合并为回归测试（`npm run test:e2e`）。

### 阶段 9：跨会话续接
用户问记忆文件在哪、要不要写 AGENTS.md、进度怎么记。我发现一个真实缺口：**`.claude` 记忆绑定在当前会话的 cwd（x-downloader），不随 wx-kit 目录走**，新会话读不到。于是把**仓库内 `AGENTS.md` 设为权威**（状态表/路线图/已定不可回退决策/命令/已知陷阱/文档索引），`CLAUDE.md` 软链到它，`.claude` 记忆瘦身成指针——单一真相源跟着仓库走。

> 心法：**连续性靠仓库，不靠会话记忆**。

## 2. 关键决策与理由

| 决策 | 选择 | 为什么（对用户/工程的影响） |
|------|------|------|
| 抓取入口 | 扫码登录后台 + 图文接口 | 代理弃用后唯一稳定的非代理路径 |
| 爬取技术形态 | 纯 Node/Electron 重写 | 去掉 Python/PyInstaller/独立 chromium/数据库——单语言单进程单二进制，打包/签名/维护都简单 |
| CLI 运行形态 | 统一 Electron 二进制双模式 | 一套能力（PDF、登录窗口）GUI/CLI 共用，零额外重型依赖；CLI 输出纯 JSON 服务于 AI agent |
| 浏览 | 内置阅读器 | 体验不跳出；md 用 react-markdown，html 用沙箱 iframe |
| 授权 | 第一阶段不做 | 开箱即用，先把核心做扎实 |
| 本地图片 | 自定义 `wxfile://` 协议 | 让相对图片正确显示，且不放开 webSecurity |
| 文章库 | 文件系统 + library.json | 第一阶段够用，便于用户直接拷走，不引数据库 |
| 进度 | download-queue 单一来源 | GUI 经 IPC 流式、CLI 走 stderr，stdout 只输出最终 JSON |

## 3. 工作流：每个 skill 把守一道关

```
分析 → brainstorming（意图/边界）→ PRD（确认）
   → writing-plans（按里程碑拆，bite-sized + TDD + 完整代码）
   → using-git-worktrees（隔离工作区）
   → subagent-driven-development（每任务：实现 → spec 评审 → 质量评审 → 修复 → 复验）
   → 端到端验证（控制者亲自跑 electron）
   → finishing-a-development-branch（合并/收尾）
   → AGENTS.md（跨会话续接）
```

每一道都是 gate：不确认不写码、不评审不算完、不验证不声称完成。这把"vibe"约束成了可重复的工程。

## 4. 评审 / e2e 拦下的真实缺陷清单

**单测全绿 ≠ 没问题。** 价值最高的发现来自两阶段评审和端到端：

**代码质量评审抓到的（节选）**
- 文件名按 UTF-16 截断会劈裂 emoji 代理对 → 改按码点截断。
- Markdown frontmatter / HTML 转义不全（换行、反斜杠、`"`）→ 补全转义。
- `library.json` 损坏会抛不可读异常；`remove()` 用 `rm -rf` 却不校验目录是否在库根内（可能误删任意路径）→ 加清晰报错 + 根目录containment 守卫。
- CLI `--help` 会污染 stdout 的 JSON 契约 → 重定向到 stderr。
- 窗口下载中途关闭 → `event.sender.send` 崩溃 → 加 `isDestroyed()` 守卫，并让下载在窗口关闭后仍跑完入库。
- 阅读器 `toWxfileBase` 兄弟目录前缀误匹配 → 加分隔符边界。
- 设置保存/库加载/删除无错误反馈 → 补 `message.error`（反馈引导行动）。

**只有端到端才暴露的 3 个硬坑**
1. **`node:sqlite` 崩溃**：cheerio 依赖 undici，其 sqlite-cache-store 在加载时静态 `require('node:sqlite')`，被打进 bundle 后 Electron 31 的 Node 没这个内置模块，主进程加载即崩。→ vite 配置 external undici（我们只用 `cheerio.load`，让它惰性永不加载）。
2. **无效文章误报成功**：微信对已删除文章返回 HTTP 200 错误页，抓取"成功"但字段全空。→ "解析后标题为空即视为无效文章"判定为失败。
3. **PDF 窗口触发自动退出**：printToPDF 的离屏 BrowserWindow 关闭时，Electron 默认"无窗口即退出"被触发，与后续的入库/输出竞速，导致进程提前终止。→ CLI 模式注册 no-op `window-all-closed`。

**e2e 自身的坑**：Antd v6 在两个汉字按钮文本间自动插空格（"阅 读"），导致 `has-text("阅读")` 匹配失败——是测试选择器问题，不是应用问题（截图佐证应用正常）。

> 心法：**对可测的逻辑 TDD，对网络/Electron/GUI 用注入+端到端**；声称"完成"前必须有证据（`naturalWidth>0`、退出码、落盘文件、截图）。

## 5. 可复用的 Vibe-Coding 心法（总结）

1. **约束先行**：先 PRD/计划/规范，再写码。没有规范的工作空间不动手。
2. **一次一个决策**：模糊需求用"带推荐+理由"的单选逐个敲定，别一次甩一堆选项。
3. **按里程碑切薄**：大项目拆成"各自能交付可运行软件"的里程碑，每个独立出计划。
4. **分子代理 + 两阶段评审**：实现与评审分离；spec 合规防过度/欠缺，代码质量防 bug。
5. **TDD 用在刀刃上**：纯逻辑 TDD；难测的（GUI/网络/Electron）用依赖注入 + 端到端断言。
6. **控制者补位**：子代理沙箱跑不了的（真实 electron），控制者亲自验。
7. **诚实压倒顺滑**：发现"AGENTS.md 没完全包含记忆内容"就直说并补齐，不糊弄。
8. **连续性写进仓库**：状态/决策/陷阱放 AGENTS.md，跟着 git 走，任何会话任何 agent 可读。
9. **干净的提交历史就是文档**：51 个语义化 commit 本身就是一份可追溯的施工日志。

## 6. 可对外分享的几个"钩子"

- "AI 写代码不可怕，可怕的是没人给它定约束。这次全程没让它放飞，每一步都有 gate。"
- "单测全绿那一刻最危险——真正的 3 个硬坑（sqlite 崩溃、假成功、PDF 截断）都是端到端才抓到的。"
- "让 AI 自己审自己：一个子代理实现，另两个子代理分别审'做对了吗'和'做好了吗'。"
- "跨会话续接别靠 AI 的记忆——它的记忆绑在目录上、不可移植。把真相写进仓库的 AGENTS.md。"
- "产品化一个原型，第一步不是写代码，是指出原型哪里是历史包袱、该砍什么。"
