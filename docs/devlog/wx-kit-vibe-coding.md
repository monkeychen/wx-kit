## 0. TL;DR

一次会话里，从"分析一个原型"开始，走完了 **分析 → 头脑风暴对齐 → PRD → 实现计划 → 分子代理执行 → 两阶段评审 → 端到端验证 → 合并 → 跨会话续接文档** 的完整产品工程闭环，交付两个里程碑。

**成果数据**
- main 上 51 次提交，干净的语义化历史（feat / fix / docs / test / chore）。
- 62 个 vitest 单测 + 1 个 Playwright Electron 端到端测试，全绿。
- 两个里程碑（M1 核心层+CLI、M2 GUI）各自经 feature 分支开发、评审、合并。
- 产出文档：PRD、M1/M2 实现计划、AGENTS.md（含状态/路线图/陷阱）、本复盘。
- **评审与 e2e 共拦下 ~20 个真实缺陷**，其中 3 个是单测覆盖不到、只有端到端才暴露的硬坑。

**一句话心得**：Vibe-coding 不是"放飞地让 AI 写"，而是**用纪律化的流程把"感觉"约束成可验证的工程**——先定约束，再分步执行，每步都有 gate。

## 1. 全程时间线

### 阶段 1：分析原型，讲真话
用户：「分析一下该项目」。
我读完 `x-downloader`（Electron + React + Python 边车爬虫 + AnyProxy 代理 + 离线授权）后，没有只夸，而是直指问题：命名与产品不符、**三套抓取机制冗余**（代理捕获 / 隐藏窗口搜索 / Python Playwright 后台爬取）、授权只签 machineId 形同"防君子"、Python 服务默认绑 `0.0.0.0`、代理崩溃后系统代理不还原会让用户断网。判断这是 PyQt→Electron 迁移的历史包袱。

> 心法：**第一性原理 + 不谄媚**。分析的价值在于指出问题与本质，而非复述代码。

### 阶段 2：产品化需求 + 约束先行
用户明确：这是原型，要产品化为 `wx-kit`（微信百宝箱，第一阶段=下载器），需求=URL 多格式下载 / 公众号批量 / 浏览 md·html / 弃用代理，新仓库位置给定，**先出 PRD 确认再下一步**。

> 心法：**约束先行**。不直接写代码，先立"要做什么、边界在哪"。

### 阶段 3：头脑风暴，把模糊需求逼成明确决策
用 `brainstorming` skill，一次一个问题（多选），敲定了几个会重塑架构的分叉：
1. 代理弃用后批量爬取入口 → **扫码登录公众号后台 + 调图文接口**（唯一可行的非代理路径）。
2. 爬取技术形态 → **纯 Node/Electron 重写，去掉 Python 边车**（去掉 PyInstaller/Playwright/随机端口/0.0.0.0 一整套包袱）。
3. 浏览形态 → **应用内置阅读器**。
4. 授权系统 → **第一阶段不做**。
随后用户追加两条需求（多 URL 输入、面向 AI agent 的 CLI），又敲定：
5. CLI 运行形态 → **统一 Electron 二进制双模式（GUI/CLI）**，让 PDF（printToPDF）与扫码登录都能复用 Electron，零额外重型依赖。

> 心法：**为决策设计提问**。每个问题的答案都改变后续架构，所以一次一个、带推荐与理由。

### 阶段 4：PRD → 迭代 → 收敛开放问题
写出 PRD（定位/目标与非目标/F1–F5/格式规格/存储结构/架构/风控/里程碑/验收），用户审阅后追加（文章管理删除+按文件名搜索、下载进度信息），再把两个开放问题敲定（search 返回候选列表、文章库目录可配置且默认用户文档目录）。

### 阶段 5：实现计划（M1）
用 `writing-plans` skill。因第一阶段体量大，**按 PRD 的 M1–M4 里程碑拆成独立计划，先写 M1**（17 个 bite-sized 任务，含完整代码、TDD 步骤、运行命令、自检）。

### 阶段 6：分子代理执行 M1
用 `subagent-driven-development`：
- 先 `using-git-worktrees` → 用户选**就地开 feature 分支**（greenfield 不必 worktree）。
- 17 个计划任务归并成 ~8 个执行单元；**每单元：派一个全新 implementer 子代理 → spec 合规评审 → 代码质量评审 → 据评审修复 → 复验**。
- 纯逻辑严格 TDD；网络/Electron 部分注入依赖 + 手动验证。
- **端到端验证我亲自跑**（子代理沙箱解析不了 electron 二进制）：真实下载产出含合法 PDF、本地化图片、库索引；去重生效。
- `finishing-a-development-branch` → 合并 main，删分支。

### 阶段 7：M2（GUI）同流程
先问 UI 栈（选 **Antd 6 + Tailwind**）→ 写 M2 计划（10 任务）→ 分子代理执行：应用壳、三个可测接缝（Settings/wxfile 协议/读文章）、IPC 桥、下载页、文章库页、阅读器、设置页。每单元两阶段评审。我做 electron 启动冒烟 + 自动化验证。合并 main。

### 阶段 8：把"手动 GUI 验证"也自动化
用户追问「不能自动化测试吗」。我用 **Playwright 的 `_electron` 驱动器**写端到端：启动真实应用、对内置 fixture 服务跑「下载→文章库→阅读器→删除→设置」，**断言阅读器图片 `naturalWidth>0`** 来证明 `wxfile://` 协议确实把本地图片喂进了页面。过程中抓到一个**测试选择器**的坑（Antd v6 会在两汉字间插空格，"阅 读"），修正后全绿，合并为回归测试（`npm run test:e2e`）。

### 阶段 9：跨会话续接
用户问记忆文件在哪、要不要写 AGENTS.md、进度怎么记。我发现一个真实缺口：**`.claude` 记忆绑定在当前会话的 cwd（x-downloader），不随 wx-kit 目录走**，新会话读不到。于是把**仓库内 `AGENTS.md` 设为权威**（状态表/路线图/已定不可回退决策/命令/已知陷阱/文档索引），`CLAUDE.md` 软链到它，`.claude` 记忆瘦身成指针——单一真相源跟着仓库走。

> 心法：**连续性靠仓库，不靠会话记忆**。

## 2. 关键决策与理由

| 决策 | 选择 | 为什么（对用户/工程的影响） |
|------|------|------|
| 抓取入口 | 扫码登录后台 + 图文接口 | 代理弃用后唯一稳定的非代理路径 |
| 爬取技术形态 | 纯 Node/Electron 重写 | 去掉 Python/PyInstaller/独立 chromium/数据库——单语言单进程单二进制，打包/签名/维护都简单 |
| CLI 运行形态 | 统一 Electron 二进制双模式 | 一套能力（PDF、登录窗口）GUI/CLI 共用，零额外重型依赖；CLI 输出纯 JSON 服务于 AI agent |
| 浏览 | 内置阅读器 | 体验不跳出；md 用 react-markdown，html 用沙箱 iframe |
| 授权 | 第一阶段不做 | 开箱即用，先把核心做扎实 |
| 本地图片 | 自定义 `wxfile://` 协议 | 让相对图片正确显示，且不放开 webSecurity |
| 文章库 | 文件系统 + library.json | 第一阶段够用，便于用户直接拷走，不引数据库 |
| 进度 | download-queue 单一来源 | GUI 经 IPC 流式、CLI 走 stderr，stdout 只输出最终 JSON |

## 3. 工作流：每个 skill 把守一道关

```
分析 → brainstorming（意图/边界）→ PRD（确认）
   → writing-plans（按里程碑拆，bite-sized + TDD + 完整代码）
   → using-git-worktrees（隔离工作区）
   → subagent-driven-development（每任务：实现 → spec 评审 → 质量评审 → 修复 → 复验）
   → 端到端验证（控制者亲自跑 electron）
   → finishing-a-development-branch（合并/收尾）
   → AGENTS.md（跨会话续接）
```

每一道都是 gate：不确认不写码、不评审不算完、不验证不声称完成。这把"vibe"约束成了可重复的工程。

## 4. 评审 / e2e 拦下的真实缺陷清单

**单测全绿 ≠ 没问题。** 价值最高的发现来自两阶段评审和端到端：

**代码质量评审抓到的（节选）**
- 文件名按 UTF-16 截断会劈裂 emoji 代理对 → 改按码点截断。
- Markdown frontmatter / HTML 转义不全（换行、反斜杠、`"`）→ 补全转义。
- `library.json` 损坏会抛不可读异常；`remove()` 用 `rm -rf` 却不校验目录是否在库根内（可能误删任意路径）→ 加清晰报错 + 根目录containment 守卫。
- CLI `--help` 会污染 stdout 的 JSON 契约 → 重定向到 stderr。
- 窗口下载中途关闭 → `event.sender.send` 崩溃 → 加 `isDestroyed()` 守卫，并让下载在窗口关闭后仍跑完入库。
- 阅读器 `toWxfileBase` 兄弟目录前缀误匹配 → 加分隔符边界。
- 设置保存/库加载/删除无错误反馈 → 补 `message.error`（反馈引导行动）。

**只有端到端才暴露的 3 个硬坑**
1. **`node:sqlite` 崩溃**：cheerio 依赖 undici，其 sqlite-cache-store 在加载时静态 `require('node:sqlite')`，被打进 bundle 后 Electron 31 的 Node 没这个内置模块，主进程加载即崩。→ vite 配置 external undici（我们只用 `cheerio.load`，让它惰性永不加载）。
2. **无效文章误报成功**：微信对已删除文章返回 HTTP 200 错误页，抓取"成功"但字段全空。→ "解析后标题为空即视为无效文章"判定为失败。
3. **PDF 窗口触发自动退出**：printToPDF 的离屏 BrowserWindow 关闭时，Electron 默认"无窗口即退出"被触发，与后续的入库/输出竞速，导致进程提前终止。→ CLI 模式注册 no-op `window-all-closed`。

**e2e 自身的坑**：Antd v6 在两个汉字按钮文本间自动插空格（"阅 读"），导致 `has-text("阅读")` 匹配失败——是测试选择器问题，不是应用问题（截图佐证应用正常）。

> 心法：**对可测的逻辑 TDD，对网络/Electron/GUI 用注入+端到端**；声称"完成"前必须有证据（`naturalWidth>0`、退出码、落盘文件、截图）。

## 5. 可复用的 Vibe-Coding 心法（总结）

1. **约束先行**：先 PRD/计划/规范，再写码。没有规范的工作空间不动手。
2. **一次一个决策**：模糊需求用"带推荐+理由"的单选逐个敲定，别一次甩一堆选项。
3. **按里程碑切薄**：大项目拆成"各自能交付可运行软件"的里程碑，每个独立出计划。
4. **分子代理 + 两阶段评审**：实现与评审分离；spec 合规防过度/欠缺，代码质量防 bug。
5. **TDD 用在刀刃上**：纯逻辑 TDD；难测的（GUI/网络/Electron）用依赖注入 + 端到端断言。
6. **控制者补位**：子代理沙箱跑不了的（真实 electron），控制者亲自验。
7. **诚实压倒顺滑**：发现"AGENTS.md 没完全包含记忆内容"就直说并补齐，不糊弄。
8. **连续性写进仓库**：状态/决策/陷阱放 AGENTS.md，跟着 git 走，任何会话任何 agent 可读。
9. **干净的提交历史就是文档**：51 个语义化 commit 本身就是一份可追溯的施工日志。

## 6. 可对外分享的几个"钩子"

- "AI 写代码不可怕，可怕的是没人给它定约束。这次全程没让它放飞，每一步都有 gate。"
- "单测全绿那一刻最危险——真正的 3 个硬坑（sqlite 崩溃、假成功、PDF 截断）都是端到端才抓到的。"
- "让 AI 自己审自己：一个子代理实现，另两个子代理分别审'做对了吗'和'做好了吗'。"
- "跨会话续接别靠 AI 的记忆——它的记忆绑在目录上、不可移植。把真相写进仓库的 AGENTS.md。"
- "产品化一个原型，第一步不是写代码，是指出原型哪里是历史包袱、该砍什么。"

## 7. M3 增补：登录 + 公众号批量爬取（CLI 优先，2026-06-07）

接续 M1/M2，第三个里程碑把"批量爬取"这个真正的差异化能力做了出来。几条值得记的：

- **范围先切，不贪大**：PRD 的 M3 含 GUI 批量页，但选了 **CLI 优先**——先交付无头核心 + CLI（agent 可调），GUI 批量页拆成 M3.5。复刻 M1→M2 的节奏，最快拿到可用能力，且核心可单测。
- **最高风险前置成 gate**：M3 唯一真不确定的是"原型 2026-02 抓的 mp 接口到 6 月还活着吗"。于是计划第 1 步就是**活体验证 spike**：扫一次码 + 调一次 `searchbiz`，肉眼确认 `ret:0`、字段名对得上，**再往下建**。证明接口活着才写后面九个任务，不白搭架子。
- **用 Electron 自带能力替代原型的重依赖**：原型用 Playwright + 独立 chromium 扫码登录（还有一大坨找系统 Chrome 的 fallback）。本项目直接用 `BrowserWindow` 载入登录页 + 监听导航抓 `token`、`session.cookies` 取 cookie——**贴合"无 Python、无独立浏览器"的既定决策**，代码还短得多。
- **纯逻辑注入 + 真实数据当 fixture**：`mp-client`/`mp-crawl` 只依赖注入的 `mpFetch`，离线可单测；测试直接拿原型 `data/articles/*.json` 的真实返回形状当 fixture。爬取层只负责"发现 URL"，落盘 **100% 复用 M1 的 `downloadArticle` + `DownloadQueue`**——列到 URL 即喂队列，去重/失败不中断/汇总全白嫖。
- **风控写进类型**：`ret=200013→RateLimited`、`200040→AuthExpired` 映射成具体异常；列表阶段命中频控指数退避（30s×3）；下载间随机 2–5s。
- **教训：fixture 必须贴真实数据形状**。两个测试当场翻车都不是实现错，是假数据失真——① 分页 fake 用 `begin/20` 索引却设 `total=4`，而真实接口 `count=20/页` 一页就返完，循环正确地停了；② `library.json` 是 `{version,articles:[]}` 包裹，我图省事写成裸数组，`.articles` 取到 `undefined`。**假数据偷懒，测试就帮倒忙。**
- **真实账号端到端收尾**：本地真登录 + 爬 2 篇（`--fakeid` 绕开"腾讯"多候选的歧义守卫）→ `succeeded:2, failed:0`，content.md/meta/本地图片全落盘。自动化测不到的真链路，靠这一步坐实。

> 一句话：**先证明外部接口活着，再写代码；能复用上个里程碑的就别重写**——M3 的下载链路一行没动 M1。

## 8. M3.5 增补：批量爬取 GUI（2026-06-07）

把 M3 的 CLI 能力接成给非命令行用户的 GUI 页。重点不在功能（功能 M3 已验），而在交互。

- **交互决策走"可视化伴侣"逐屏定**：开了浏览器版伴侣，把三个关键交互做成线框让安哥点选——① 单页渐进 vs 分步向导 → 选单页；② 整页登录引导 vs 顶部状态条+禁用 → 选整页引导（没登录就只给"去登录"一件事，不摆点不动的控件）；③ 实时逐篇列表 vs 极简进度 → 选实时（慢任务越透明越安心）。**比纯文字描述快得多，也避免我猜错方向白写。**
- **核心层只开两个小口子，其余全接线**：`DownloadQueue` 加可选 `shouldContinue`（取消，URL 下载页不传、行为不变）；`mp-crawl` 加 `onListed`/`onItem` 逐篇上报。爬取/落盘逻辑一行没动 M3。
- **"列表先铺、逐篇翻状态"**：crawl 一拿到 `appmsg` 列表就把全部标题整批推给渲染层立刻铺行（"等待"），再逐篇翻 ⟳→✓/⊘/✗。ETA 纯前端用"已完成/已耗时"估，不进后端。
- **取消=停后续、已下保留**：和去重续传天然兼容，省掉了"暂停/恢复"这套状态机（YAGNI）。重试单篇直接复用现有 `download` 接口，不造新东西。
- **安哥的一句话改了测试策略**："session 落盘后过期前你完全可以自动化"。于是把"真链路全靠手动"降级为"半自动"——验证脚本把本地缓存的 session 拷进隔离 userData，Playwright 驱动真实 GUI 搜号 + 爬 2 篇，断言实时行变"成功、成功"，全程无人扫码。只有 session 失效才需人工。**凭证留 userData（本地不同步）而非库根，是安全与"统一可配置"权衡后的选择。**
- **教训延续**：两个测试当场翻车仍是 fixture 失真（分页 begin/20 设错 total、library.json 包裹格式），不是实现错——假数据偷懒，测试就帮倒忙。

> 一句话：**UI 的"形"用可视化伴侣和安哥对齐，"魂"（取消/重试/实时）靠把核心层的两个小钩子接出来**——大头还是复用，不是重写。

## 9. M4 增补：electron-builder 打包（2026-06-07）

第一阶段最后一块拼图——把 wx-kit 打成可分发的 mac/win 安装包。

- **签名/公证先不做**——未签名包够用且贴「开箱即用、不加门槛」的阶段定位。代价是首次打开有系统警告，靠 README 写清放行步骤（mac 右键→打开 / `xattr -cr`；win SmartScreen→仍要运行）。"反馈引导行动"原则在这里延伸到了分发环节。
- **图标先截图签收、再定代码**——"宝盒"这个概念一开始就交给安哥过目再实现：先渲染了第一稿（拱盖像侧视、视角打架），安哥一指出，我重画成"箱体等宽的正面拱顶 + 金属带贯穿"，再渲染、确认、才提交 `build/icon.png`。**产品脸的决策权在用户手里，不在 AI 的猜测里**。

## 11. M6 增补：下载闭环 + 下载历史（2026-06-08）

把"能下载"补成"下完能确认、能找回"。两个需求 R1（就地确认/阅读）+ R2（历史）被**合并成一个组件**而非各做一套。

## 12. M7 增补：反馈引导——把"静默"点亮成"看得见"（2026-06-08）

R5 的本质不是加功能，是把系统已有但藏起来的状态**翻译给用户**。core 早就在频控时退避 30/60/90s、早就抛出各种错误，只是 GUI 把这些咽了。M7 做两件事：让退避可见、让失败可行动。

- **先确认 bug 再动手**：这个里程碑由安哥一个真实 bug 引出——「猫笔刀 2026-05-24~27 下载 0 篇」。没有猜，写了 spike 打真实 API，实测出 `appmsg` 每页只回 5 篇而代码 `begin += 20`（固定步长）每页跳过 15 篇。TDD 用「5 篇/页」模型复现并锁死，`begin += pageLen`（原始返回篇数）修复，真实 API 验证窗口正确返回 2 篇。**频控/分页这类外部契约，实测一次胜过读十遍文档。**
- **可见的退避，不是进度条**：退避是"在等"，不是"在做"，所以用独立朱砂横幅 + 客户端每秒倒数（「约 N 秒后重试 · 第 k 次」），非报错红——传达"系统在替你扛频控、还要等多久"，而非"出错了"。core 加一个 `onBackoff` 回调上报，IPC 透传 `{kind:'backoff'}`，渲染端拿 `until` 自己倒数。**反馈要引导行动（等着就好），不是只报状态。**
- **错误话术归一，且放对层**：原始 `fetch timeout after 20000ms`、`invalid or unavailable article` 对用户是噪音。`explainError` 把它们按 code + message 关键字两路归一为「人话标题 + 下一步」（登录过期→去登录、网络超时→查网络、文章无法访问→可能被删），原文折叠在 tooltip 备查。**放哪一层是第一性问题**：它是纯展示逻辑，CLI 契约是 JSON 给 agent 解析、本就不要人话——所谓"core 共享"是 YAGNI，放 `src/renderer/error-explain.ts` 才不破"renderer 绝不 import core 运行时"的边界（现有 renderer→core 全是 `import type`，我差点成为唯一的运行时 import）。
- **验证戳破了一个伪需求**：原计划给"完成/取消后回到配置"加「返回配置」按钮（R3 残留）。真实 session 截图时按钮的 testid 等不到——因为 `await mpCrawl` 完成即 `setRows([])`、配置卡本就自动重现，按钮只在 done 事件到 promise resolve 间闪现 200ms。**端到端验证当场证伪了一个我以为要做的功能**，撤掉，R3 标记"既有流程已满足"。
- **逼出隐藏元素也要看真实渲染**：退避横幅和失败行话术要真实 200013/失败才出现，没法按需触发。没有用"逻辑测过就算了"糊弄——临时种态（改初始 state→build→截图→还原）把两个元素逼到真实渲染里看了一眼确认无误，再还原。遵守 M5 立的规矩到底。

> 一句话：**M7 把"后端在忙什么"老实告诉用户**——退避不再像卡死，失败不再甩英文报错。最贵的一课来自一个真实 bug 和一次证伪：**实测胜过臆测，验证能反过来砍需求。**

## 13. M8 增补：PDF 防跨页切断（R4，2026-06-08）

一行需求「PDF 里图片别被页边界切断」，做法上有两条路，选型决定了它放哪、影响谁。

- **`@media print` 注入 HTML，而非 `insertCSS` 注入窗口**：PDF 走的是 `printToPDF` 加载保存好的 `index.html`。把防切断规则作为 `@media print` 加进 `buildHtml` 的内联样式——随 HTML 走、只在打印态生效（屏幕阅读器零影响）、用户自己 Ctrl+P 打印这份 HTML 也受益，还能在纯函数 `buildHtml` 层 TDD，让依赖 electron 的 `export-pdf.ts` 一行不改。比起在离屏窗口 `insertCSS`（PDF-only、HTML 不沾），前者自包含、可测、红利更多。
- **验证戳破了自己的假设**：本想用「一张高图」做有/无规则对照，结果发现 **图片是「替换元素」，Chromium 默认就把放不下的整张图下移、根本不切**——裸图对照看不出差别。真正会被默认拦腰切断的是**代码块/表格**（普通块容器，默认随意分页）。于是把对照换成一段会跨页的 `<pre>`：无规则版切到第 17 行、有规则版整块下移次页完整。**做验证时别假设 bug 长什么样——亲手把它复现出来，假设常常是错的。**
- **端到端兜底**：e2e 默认格式不含 pdf，R4 没有自动化像素断言，就用「真实文章 CLI 出 PDF + 用 poppler 渲染翻页目视」端到端兜——这类依赖 electron/Chromium 的保真问题，单测断不了像素，眼睛和真实出件才是终审。

> 一句话：**M8 最大的收获是验证而非编码**——一行 CSS 规则，却靠「亲手复现 bug」纠正了「图片会被切」的错误直觉，定位到真正受益的是代码块/表格。

## 14. M9 增补：文库组织 + 卡片/列表视图（2026-06-08）

把文库从「只能搜+删」做成可治理的藏馆：排序、按公众号筛选+分组、批量删除；安哥又追加了「访达式列表视图」。

### M9 后：把 e2e 扩成 M5–M9 全功能回归套件，顺手逮到真 bug

安哥要「所有功能点都测过去、完善 e2e」。把原本只覆盖「URL 下载→阅读→删」的 e2e 扩成 24 条断言的多文章回归套件（夹具跨 2 公众号 + 一篇无标题失败页），覆盖：历史就地确认/复制下载项/已存在跳过/失败重试、设置库根、文库排序/筛选/分组/卡片⇄列表/双击阅读/批量+单篇删除、真实公众号抓取。过程踩了三个有价值的坑：

- **e2e 逮到一个真实 UX bug**：双击列表行进阅读总是失败、只选中 1 篇。根因不是测试写错——是**单击选中后批量条插入、把列表向下挤**，双击的两次点击跨越了这次重排、第二下点偏了。**真人双击同样会中招**（两次点击落在同一屏幕位置，但内容已下移）。正解不是改测试，是把批量条改成**底部浮动条**（不再 reflow）。自动化测试替我发现了肉眼演示时没注意的交互缺陷。
- **夹具按路径区分文章**：`articleId` 回退按 `origin+pathname` 归一、**忽略 query**，于是 `?id=a1/a2` 全撞成一篇——必须用不同**路径** `/article/a1`。这恰好暴露了「断言只数历史行数、没数文库实际篇数」的盲点：3 行里其实 2 篇是跳过的。
- **antd v6 的两个选择器坑**：Select 的可点根是 `.ant-select`（不是 v5 的 `.ant-select-selector`）；按钮文本被自动插空格成「删 除」，`has-text("删除")` 匹配不到、得用单字「删」（这条 AGENTS.md 早记了，这次又踩）。

> 一句话：**e2e 不只是回归网，还是 bug 探测器**——它替我发现了「选中即重排」这个肉眼演示漏掉的真交互缺陷，催出了底部浮动批量条。

## 15. v0.2.1 增补：依赖安全审计——把 26 个 Dependabot 清零（2026-06-09）

v0.2.0 发布后 GitHub 报 26 个 Dependabot 漏洞。一个看似「升级版本号」的小活，真正难点全在**国内网络环境**，逼出几条可复用的工程教训。

- **先看清「真漏洞」再动手**：26 个全是 `package-lock.json` 的**间接依赖**。真能动手的只有直接依赖：electron / vite / vitest / electron-builder。Round 1 先升 vite v6 + vitest v3 + electron 38（每个 bump 后跑 119 单测 + tsc + lint + e2e），收掉 16 个。剩 10 个（electron 3 + tar 6 + glob 1）要跨大版本。
- **「单测全绿 ≠ 升级安全」，打包正确性靠真实启动**：electron 跨大版本（31→38→42）最怕 `undici external` 失效（cheerio 间接依赖 sqlite-cache-store 静态 `require('node:sqlite')`，打进 bundle 会让主进程加载即崩）。e2e 走的是 `vite dev`、不覆盖打包后的 .app。所以每次大版本跳，都**真实启动打包后的 `wx-kit.app` + 触发 cheerio 解析路径 + 断言零 console/page 错误**——这是 M4 立的规矩，在 electron 42 上再次兑现。
- **最大的坑是网络，不是代码**：本机常驻死代理 `http_proxy=8118`。`@electron/get` 用 Node 24 内置 `fetch`(undici)，它**默认不读 `http_proxy`**、直连 github 在国内不可达 → `UND_ERR_CONNECT_TIMEOUT`；走 8118 拉 130MB 又**中途截断**。**正解：electron 二进制走 npmmirror 国内镜像 + 给镜像域名加 `no_proxy` 直连**（cdn.npmmirror.com ~1.5MB/s 稳），代理保留 set。安哥据此立了项目规则：npm/二进制走国内镜像优先、代理只兜底。
- **坑中坑**：本机**没有 `timeout` 命令**——`timeout 12 curl …` 静默报 command-not-found 让 curl 根本没跑，误判「镜像 0 字节不通」，差点把唯一正解否掉。验证下载只能用 `curl --max-time`。
- **方向反转的第二条规则**：二进制下载要镜像+保留代理，但 **`gh`/`git push` 到 github 反而要 unset 代理**——8118 代理传 github 大文件同样卡死（单个 140MB release 资产传 10+ 分钟零字节），直连才稳。安哥实测后补进规则。
- **证伪一次冒进**：升级中途一度想「electron 42 + builder 26 全跟 latest」，但在「代理拉不全二进制」时我误判为「本机环境物理不可解、回滚」。其实是镜像方案没用对（`timeout` 坑 + 没配 `ELECTRON_MIRROR`）。**环境问题也要找根因，别急着判死刑**——换镜像后一次过。

> 一句话：**这次没写几行代码，难的全是「让正确的包从正确的源、用正确的代理策略、完整地下到本地」**。最后 28 个 Dependabot 全部 `fixed`、0 残留 0 忽略（升级中 GitHub 又新录 2 条 electron advisory 一并被 42 覆盖）。教训沉淀成项目规则（AGENTS.md）+ 记忆，比清零本身更值钱。

## 18. M11 公众号订阅：单进程桌面应用里做「类 RSS」（2026-06-16）

v0.3.0 的大头。难点不在写代码，在**在「单进程、无后台、会过期的登录态」约束下，把 RSS 那套订阅语义做得诚实**。

- **设计阶段就被一条数据约束改了形**：已下载文章只存公众号名、不存 `fakeid`，而轮询新文章必须有 fakeid。fakeid 只在「按公众号抓取历史」和「搜号结果」里有 → 订阅列表只能由这两处派生，纯 URL 下载来的号进不了。这条在 brainstorming 读代码时就发现并写进 spec，避免了实现到一半才撞墙。
- **把「会动的」全部纯函数化、注入化，副作用留在最外层**：核心三件套全部可单测——`subscriptions`（存储 + `accountsFromHistory`/`mergeAccounts` 派生）、`subscription-schedule`（`shouldCheckNow` 纯判定，覆盖「到点触发/已检查跳过/启动补检」各分支）、`check-subscriptions`（注入 `listFn`，断言新检测/水位推进/频控退避/单号失败隔离/登录失效中止）。主进程的 scheduler 与 IPC 只做「读配置 → 调纯判定 → 触发 → 写盘 → 发事件」的编排，不含可分支的业务逻辑。17 条新单测，主进程接线靠 build + e2e。
- **诚实约束写进代码也写进文案**：定时**只在 app 开着时生效**——这是单进程桌面应用的物理事实，不假装有后台。`scheduler.start()` 立即 tick 一次做「启动补检」（当天时刻已过且未检查就补一次），`shouldCheckNow` 的「lastCheckedAt < 今天 slot」分支专门覆盖它；订阅页文案直说「关闭时错过的检查会在下次启动补做」。登录态会过期 → `checkSubscriptions` 撞 `MpAuthExpired` 整体中止、不静默吞，主进程置 `subsAuthExpired` 标志、订阅页弹登录引导横幅。
- **水位语义防「存量当新文章」**：订阅那一刻把水位设为该号当前最新文章的 `createTime`（取不到就用「现在」秒），只有之后发布的才算新；每次检查后水位推进到本次最新。否则一订阅就把人家几百篇历史全标成「新」砸出来。
- **计划里给 UI 留口子、执行时按真实代码接入**：`Settings.tsx` 的写法我写计划时没读，于是计划只说「按其既有 state/persist 模式接入、控件代码给全」。执行到那步先读——发现它是「单个 `s` 状态 + 一个『保存设置』按钮统一持久化」，而非我设想的逐控件 `persist()`。按真实模式接（`setS({...s, field})`），没硬塞第二套状态。**计划对未读文件的假设要标成「待对齐」，别写死。**
- **e2e 在不依赖网络的边界上验**：订阅检查要真 session，e2e 聚焦「不需要网络的部分」——导航项位置、订阅页可达/渲染（列表或空态）、检查更新按钮存在、设置三控件存在。恰好 e2e 的 account-mode 用真实缓存 session 抓了一个号，于是订阅页真的走了一遍「从历史派生账号」的活路径，零 console 错误。
- **收尾自动化**：143 单测（+17）全绿、e2e 全绿后自动合 main、删 `feat/m11`、八个任务逐一 commit；PRD §4 R2 九条验收逐条勾，并诚实标注「真实壁钟到点的整链 e2e 测不了、由构成逻辑单测兜底」。

> 一句话：M11 的价值是示范了「**约束驱动设计**」——单进程 + 无后台 + 登录会过期这三条物理约束，决定了订阅必须是「opt-in 的、app 开着才跑的、过期要明示的、启动会补检的」，而不是照抄服务端 RSS。把这些约束写进 `shouldCheckNow` 的分支和页面文案里，比假装它是个常驻服务诚实得多。

## 19. M12 订阅触发升级 + 检查可观测性（2026-06-16）

安哥用了 M11 订阅后两问：① 能不能按小时而非只能按天查？② 怎么知道定时真跑过？前者扩触发机制，后者补可观测性。一次里程碑两件。

- **「间隔从何时起算」用户选了网格锚 0 点，我照办但留了诚实约束**：每隔 N 小时 = 每天从 0 点切 N 小时网格（N=6 → 0/6/12/18）。N 不整除 24 时末段不足 N 小时，当天在次日 0 点收口重置——`nextScheduledInstant` 用 `min(候选, 次日0点)` 处理，单测专门覆盖 N=5 now=22:00 → 次日 0 点。没假装能精确均匀分布，这是单进程桌面 app 的事实。
- **判定统一成一句抽象**：`lastScheduledInstant(now, config)` = 当前已过的最近计划时刻；`shouldCheckNow` = 该时刻存在且 `lastRunAt < 它`。daily 在今天时刻前返回 null，interval 恒有最早当天 0 点。这一个抽象同时涵盖了 daily/interval 两种模式 **和** 启动补检（app 没开着错过某网格点，启动时 now 已过该点且 lastRunAt 早于它 → 补一次）——三件事一个函数收口，比写三段分支干净。
- **可观测性的关键判断：日志放哪**。放文库目录会污染用户浏览的内容、且那是「下载产物」的归属；放 `userData`（与 `mp-session.json` 同处）是「诊断数据」的归属。所以 `subscriptions.json` 只留最近 50 条驱动页面、全量落 `userData/subscriptions-check.log`。**数据按语义归类，不按技术层堆放。**
- **写盘失败不阻断主流程是硬约束**：`logCheck` 整体 try/catch——日志是「锦上添花的留痕」，磁盘满/权限错时不应让订阅检查本身失败。这条直接对应 spec 的错误处理段，不是事后补的。
- **触发器参数化（auto/manual）一笔带过却关键**：`runSubscriptionCheck(trigger)` 让定时走 `'auto'`、手动「检查更新」走 `'manual'`，检查记录里能一眼区分「这是我点的」还是「定时自己跑的」——这正是回答安哥「怎么知道定时生效」的直接证据：开了自动检查、到点后检查记录冒出 `auto` 条目。
- **文档全程同步、不是事后补**：安哥专门叮嘱「记得更新 PRD 等」。M12 从设计到合入，PRD R3 需求 + §4 验收七条逐条勾、ROADMAP 状态与索引、devlog §19 一起进。每个里程碑交付 = 代码 + 文档对账，不留「文档漂移」债（呼应 §16 的教训）。

> 一句话：M12 的方法论是**「把会动的判定抽成一个纯函数、把留痕和主流程解耦、数据按语义归类」**——daily/interval/启动补检三种触发语义被 `lastScheduledInstant` 一个抽象吃掉，日志写盘失败被 try/catch 隔离不连累检查，诊断日志归 userData 而非文库。单进程桌面 app 的约束照样能把「类 RSS」做得可见、可控、诚实。

## 20. UI 一致性打磨：四个页面栏宽统一（2026-06-16）

v0.3.0 收尾后安哥扫了一眼 UI：下载、设置两页左右留白大，订阅、文库两页却贴边——切页面时内容栏宽度跳变。一次小打磨。

- **方向让安哥定，给推荐**：统一成满宽还是统一成居中窄栏，是观感取舍（文库的卡片网格/列表表吃宽度，表单怕散）。安哥选满宽——四个页面去掉 `.page-narrow`、都只留 `.page` 的 40px 内边距。一刀切齐，最省心。
- **视觉一致性用测量验证，不靠肉眼**：截了四个页面的图后，没凭感觉判「看起来一样了」，而是用 Playwright `getBoundingClientRect` 量每个页面 `.fade-in` 的 left/width——改前下载/设置 width=860 居中（左右各空 ~170）、订阅/文库满宽；改后四个全部 left=40 满宽起步。**「看起来一致」是主观，「left 全等于 40」是客观**。视觉回归该尽量量化。残留的 11px 宽度差（1109 vs 1120）也查清了：是内容多寡导致有的页面出了滚动条，不是栏框不一致。

## 21. 频控韧性 + 去规律化：从「治标」到「治本」（2026-06-18~22）

安哥连发几问把频控这条线逼到了根上：① 按公众号下载时频控 retry，点取消没用；② 订阅被限时还在 retry 吗，有就取消；③ 订阅太规律，容易被识别，怎么优化？三问串成一条从「让取消生效」到「别火上浇油」再到「别长得像机器」的递进。

- **取消失效的根因是「信号够不着」，不是「按钮没接对」**：用 systematic-debugging 定位——按公众号下载分列表/下载两阶段，取消信号只在下载阶段的 `DownloadQueue` 循环开头被查；而频控 retry 发生在**列表阶段**的 30/60/90s 指数退避 `sleep` 里，那段干等根本没看取消标志。所以点了取消、标志也置上了，但代码正卡在 sleep 里，要等退避跑完（最坏 ~3 分钟）才理你。修法：串入 `AbortSignal`，退避等待用 `abortableWait` 让「sleep 与取消竞速」，一取消立即返回；IPC 从 `cancelRequested` 布尔换成 `AbortController` 让信号能传进退避。`mp:crawl:cancel` 频道名不变 → 渲染层取消按钮一行没改。**先写「卡在退避里点取消」的失败测试（模拟永不 resolve 的 sleep）复现卡死，再修。**
- **订阅的频控 retry 是「火上浇油」，直接删而非优化**：订阅检查命中频控原先也退避重试 3 次。但回到第一性问题——重试是在**已经被限的登录态**上继续追加请求，只会刷新/延长冷却窗口，是反效果。所以不是「优化退避曲线」，是**根本不该重试**：命中即把该号记为本轮失败、跳过，等下一轮定时检查自然再来。连带删掉因此变成死代码的 `onBackoff`。**正确的频控应对是「停手等冷却」，这恰好也是 ①「取消能即时停」的价值所在。**
- **去规律化要分清「真指纹」和「治标剧场」**：先把当前的机器指纹列清楚——每天同一秒触发、固定 fakeid 顺序、**恒定 2.0s 账号间隔**。三个里恒定间隔最像机器（人不可能分毫不差 2 秒）。对症加抖动：间隔 `2000→randMs(3000,8000)`、触发时刻叠加确定性顺延、账号顺序每轮打乱。**但我对安哥说了实话：抖动削的是「明显机器节奏」这个把柄，挡不住高频轮询本身——微信频控第一身份是登录态（账号+token+cookie）+ 请求量/频率，IP 和节奏是次要信号。真正的大杠杆是降频率/减账号数。抖动是锦上添花，不是免死金牌。** 不夸大疗效，是这条线最该守住的诚实。
- **「确定性抖动」是 B 的关键设计**：触发时刻的随机顺延**不能每次轮询重算**——否则 `shouldCheckNow` 会抖成早触发/重复触发。做法：偏移由「时段起点」经整数哈希确定性派生（同一时段恒定、日间/段间各异），只往后顺延 0~30min（不会早于设定点）。且**现有网格函数一行没动**（旧测试全绿），新增 `scheduleJitterMs`/`shouldRunCheck`（在 `shouldCheckNow` 上叠抖动闸门、去重仍按原始时段 key 防双触发）/`nextCheckAt`（显示用），scheduler 与页面「下次预计」切到新函数。**加特性靠新增纯函数包裹，不改既有契约。**
- **「砍 count 减频控」被当场证伪**：安哥曾提议把订阅回看篇数从 20 砍到 3 减频控。回到机制——`listArticles` 单页 `pageSize=20`，count≤20 时**一页拿完、不翻页**，所以 count=3 和 count=20 对微信都是**同 1 次请求**，频控压力一样；而 20 是「回看窗口」，砍小了某号一次群发超过窗口数就会**永久漏文章**。结论：**零收益、纯增风险**，保持 20。这条已沉淀进 memory，免得日后又被当频控手段重提。

> 一句话：频控这条线的方法论是**「先找信号够不着的根因（取消），再回第一性删掉反效果的重试（别火上浇油），最后对真指纹加抖动且不夸大疗效（去规律化）」**——三步都用失败测试先复现、用确定性设计避免引入新抖动 bug、用机制分析证伪「砍 count」这类想当然的治标。

## 22. 阅读器 Markdown 标题重复：修在「谁该负责」那一层（2026-06-22）

安哥发现文库里读文章，md 视图标题出现两次。一个小 bug，但定位修在哪一层有讲究。

- **根因是两处都「尽职」地渲染了标题**：`buildMarkdown` 在正文顶部注入 `# <title>`——这是对的，导出的 `content.md` 用 Obsidian/编辑器打开时需要标题。而阅读器自己也渲染了一套标题头（kicker + h1 + byline），于是 ReactMarkdown 把正文那行 `# <title>` 渲成第二个 h1。**不是谁写错了，是文件态与阅读态对「标题」的需求不同。**
- **修在表现层，不动磁盘文件**：.md 文件该留标题（对外部工具有用），重复只发生在「阅读器既画了自己的标题头、又渲染了带标题的正文」。所以让阅读器在渲染前剥掉那行——新增纯函数 `stripLeadingTitle`，只在「开头第一处非空内容恰为 `# <title>`」时剥离，正文里真实的同名小节、非首行标题都保留。7 条单测覆盖（含 CRLF、前导空行、特殊字符标题、非首行不剥）。**fix 选在「制造重复的那一层」，而非去改对外部正确的文件。**

> 一句话：同一份内容在「文件态」和「阅读态」的诉求可以不同——标题该留在文件里，重复该消在渲染前。修 bug 先问「谁该为这个症状负责」，别图省事去改一个对其他消费者正确的产物。

## 23. undici 安全补丁：先评暴露面、再升、最后验「external 陷阱」（2026-06-22）

push 后 GitHub 弹出 2 个 Dependabot 漏洞（1 high + 1 moderate），安哥让先分析。一次「告警≠风险」的处置示范。

- **先回答「我们到底受不受影响」，而不是条件反射升级**：两个告警同一个包 `undici@7.27.1`（被 cheerio 与 `@electron/get` 共用），都在 7.28.0 修。逐条核对暴露面——#32 需要用 undici 的 SOCKS5 `ProxyAgent`（我们没有）、#33 需要用 undici 的 HTTP 缓存（我们没有，而且 undici 已 `external` 永不加载）；自有代码零直接引用 undici，只用 `cheerio.load`（纯内存解析，从不走 cheerio 那条 undici-backed 的网络抓取 API）。结论：名义 high/medium，**本项目实际暴露面为零**。**Dependabot 按版本号判定，不按调用路径——所以「分析」这一步是把『被标记』翻译成『真实风险』，别把扫描器的版本匹配当成事实暴露。**
- **升级的理由是「保持归零 + 白送的修复」，不夸大成救火**：暴露面虽零，但照 v0.2.1（§15）把 Dependabot 清零的既定方针 + form-data（`d39b51e`）的 override 先例，加 `overrides: { "undici": "^7.28.0" }`。全局 override 把三处实例（cheerio `^7.24.4`、@electron/get `^7.19.0`、node-gyp `^6.25.0`）统一 deduped 到一份 7.28.0——node-gyp 那份从 6.26.0 被抬到 7.x 是构建期工具，靠下面的 build 验证兜底。**对外说清「这是卫生维护，不是漏洞被利用了」，是这条最该守的诚实（呼应 §21 不夸大抖动疗效）。**
- **transitive bump 后必须重验「undici external 陷阱」**：项目有条硬陷阱——cheerio 依赖的 undici 里 sqlite-cache-store 静态 `require('node:sqlite')`，Electron 内置 Node 没这个模块，一旦被打进 bundle 主进程加载即崩；我们靠 vite `external` 让它惰性永不加载。undici 版本一变，这条必须重测，`npm install` + 单测不够。验证四连：174 单测 + tsc + lint 绿 → mac build 成功 → grep `main.js` 确认 undici 没被打包（`node:sqlite` 0 次、`require("undici")` 仅 1 处惰性引用）→ 真实启动打包后的 `.app` CLI（`auth-status` exit 0、合法 JSON，主进程加载不崩）。**「升级没破坏 external」这种不变量，要用打包产物 + 真机启动证明，不能靠『版本号兼容』推断。**

> 一句话：依赖告警的正解是「先把『被版本匹配标记』还原成『真实调用路径暴露』再决定要不要慌，升级当卫生维护诚实陈述，并对项目特有的不变量（undici external）用打包+真机启动重新验证」——三步都呼应同一条原则：别让工具的判定替代你对自己系统的判断。

## 24. v0.4.0 启动 + M13 存储加固：subagent 驱动开发首秀（2026-06-22）

安哥提了两个深问题——① 文件存储这套机制会不会有问题？② 能不能把文库文章当素材喂给本地 agent 做选题/写作/审阅？这把 v0.4.0 的方向定了下来：**文库供料 agent + 先夯实存储地基**。本节记两件事：v0.4.0 怎么从「问题」收敛成 PRD，以及 M13 第一次用 subagent-driven-development 执行计划的体感。

- **存储隐患是「读出来的」，不是「猜出来的」**：回答「文件存储有没有问题」时没泛泛而谈，而是把 `Library`/`History`/`Subscriptions` 三个索引类的读写真读了一遍，定位到两个真实隐患——① `writeFile` 整文件覆盖**非原子**，崩溃/断电留半截 JSON → 整个索引报废；② 每次写是 `read→改→write` 中间有 `await`，**后台订阅自动下载与前台操作并发会丢更新**（已开自动订阅，这并发是现实，不是理论）。还发现每篇文章目录都有 `meta.json` → `library.json` 本质是**可重建的派生索引**，于是把「损坏只能手删重来」变成「一键 rebuild」。**「存储会不会有问题」这种问题，答案在代码里，不在直觉里。**
- **架构边界先于功能定**：供料这条线最关键的决策不是「导出什么格式」，而是「编排放哪」。brainstorming 里把它逼成单选——wx-kit 只供料、编排活在外部 skill（复用 khazix-writer/hv-analysis），而不是在 app 里造「创作模块」。**守住「下载器」本分，不让它膨胀成「内容工作室」**——这呼应了项目「百宝箱但不预造空模块（YAGNI）」的既定决策。素材用途也写进 PRD §6：选题/研究/启发，不洗稿、强制人在环中。
- **subagent-driven 的价值在「上下文隔离 + 双道闸」**：M13 拆成 8 个 bite-sized 任务，每个派一个**全新 subagent**（带完整 task-brief，不继承主会话历史），完成后**独立 reviewer**（spec 合规 + 代码质量两道验verdict）。模型按任务挑：新文件+完整代码用便宜档（haiku 转写），改既有文件/并发原语用 sonnet 评审。**fresh context 让每个实现者只盯自己那点事，主控（我）只做编排，上下文不被污染。**
- **评审与实现者都逮到了我计划里的真 bug**：Task 4 的实现者发现**我写的并发 append 测试漏了 `now=` 参数**——默认 `Date.now()` 会把日期 1972 的测试事件按保留期裁掉，导致加锁也变不绿。它正确地补上 `now=1000*DAY`，并标 DONE_WITH_CONCERNS 上报。**计划再细也会有洞，双道闸（实现者自审 + 独立评审）就是网。** Task 1 的评审还揪出原子写失败时不清理临时文件（会在库根留孤儿 `.tmp-*`），补了 best-effort 清理。
- **并发丢更新这种隐性 bug，用「并发测试」逼出来**：每个接锁的任务都先写「两个实例并发 add/append」的失败测试——RED 阶段真的只剩一条记录（丢更新复现），加锁后 GREEN 两条都在。**抽象的「竞态」要落成一个会红的测试，才算真的被钉死。**
- **子 agent 跑不了 electron，真机验证留给主会话**：Task 7 的 GUI 接线（IPC+preload+api+设置页按钮）子 agent 能写能过 tsc/lint/单测，但**沙箱解析不了 electron 二进制**，真机验证（`npm run test:e2e` 实机驱动全流程）由主会话做——36 项全过、0 console 错误，证明原子写+写锁垫在底下后实机读写没坏。**知道哪步必须回主会话做，是 subagent 编排的一部分。**

> 一句话：M13 的方法论是**「隐患从代码里读、边界先于功能定、用 fresh-context subagent + 双道闸执行、把竞态钉成会红的测试、电不了的真机验证留主会话」**——第一次 subagent-driven 跑下来，最大的收获是双道闸真能逮出连计划作者都没看见的 bug。

## 25. M14 供料能力：把文库变成 agent 的「素材出口」（2026-06-22）

M14 给 v0.4.0 的主线落地第一块：让选中的文库文章变成 agent 能直接吃的结构化素材。沿用 M13 的 subagent-driven 节奏，4 个任务一气呵成。

- **CLI 与 GUI 共用同一套核心，清单同源**：选料过滤（`selectArticles`）+ 清单组装（`buildManifest`）+ 写盘（`writeMaterialExport`）全在 `material-export.ts` 一处，CLI `library export` 走 stdout、GUI `library:exportMaterial` IPC 写文件，但**清单由同一个 `buildManifest` 出**——不会出现「命令行导的和界面导的字段不一样」这种漂移。这正是「系统承担复杂性」：两个入口、一套真相。
- **只给路径不内联正文**：清单字段固定 `{id,title,account,author,publishTime,sourceUrl,dir,contentPath}`，`contentPath = dir/content.md`。stdout 轻、agent 按需读盘，复用磁盘已有的 `content.md`，不把正文塞进 JSON。**供料的边界感：wx-kit 给「料在哪」，不替 agent 决定「怎么吃」。**
- **「无选料器报错」是防误操作的小闸**：`library export` 不给 `--ids/--since/--account` 也不给 `--all` 时直接 `NO_SELECTOR` 报错退出，而不是默默导全库——避免手一滑把整个文库吐出来。要全库必须显式 `--all`。**危险的默认行为，宁可报错也不猜。**
- **`exports/` 的目录归属在 M13 就埋好了**：GUI 导出写到库内 `exports/<时间戳>.json`，而 M13 的 `rebuildLibrary` 递归扫描早已忽略 `exports/`——所以素材清单不会被误当成文章重建进索引。**跨里程碑的约定提前对齐，比事后打补丁省事。**
- **复用 M13 的原子写**：`writeMaterialExport` 用 `atomicWriteFile` 落盘，导出文件天然也享受「写一半不留半截」。地基夯实后，上层白捡。
- **GUI 按钮镜像既有批量删除**：文库批量条加「导出为素材」，处理器结构、错误提示、`api.reveal` 点击跳转全照 `batchDelete` 的样子——新入口长得像老入口，用户不用重新学。真机验证（`npm run test:e2e` 全流程 + 0 console 错误）由主会话做（子 agent 跑不了 electron）。

> 一句话：M14 的方法论是**「一套核心两个入口（CLI/GUI 同源）、供料只给路径不越界、危险默认宁可报错、跨里程碑约定提前对齐」**——把文库变成素材出口，难的不是导出本身，是守住「下载器只供料」的边界不让它膨胀。

## 26. M15 贯通样例 skill：编排自己的 skill，先读清它们到底是什么（2026-06-22）

v0.4.0 收口：交付一个仓库内的 Claude Code skill `agent/wx-kit-compose`，把 M14 的「素材出口」接到创作流水线——选料 → 选题 → 写作，两个人工检查点。这一节的教训不在编排本身，在「动手编排别人的 skill 前，先读清它们真实是什么」。

- **不套 M13/M14 的执行法，因为交付物变了**：M13/M14 是 TDD 应用代码，配 bite-sized 计划 + subagent 执行。M15 交付的是**一个 skill（markdown 指令）**——没有 red/green 单测、拆不成独立 subagent 任务、验证必须在主会话跑真实素材 + 调别的 skill。硬套 TDD 计划是形式主义。识别「这个里程碑的交付物是什么形态」，再选执行法,而不是惯性套上一个。
- **spec 写「复用 hv-analysis」，读了代码才发现错位**：设计稿当时顺手写「选题候选(复用 hv-analysis)」。真去读 `hv-analysis/SKILL.md` 才看清:它是**重量级单主题联网深研 → 出 PDF 报告**的工具,不是「扫几篇文章提选题方向」的轻量助手。把它整个塞进选题步会变成跑一份 PDF 研究,完全错位。**这是「写文档时想当然、没核对被引用对象真实形态」的又一次复发**(同 §22 fakeid——spec 说有、代码里没有)。于是把发现摆给安哥,他定:选题轻量内联、hv-analysis 仅作旁路。**编排别人的能力前,先把那些能力读穿——description 之外要看它实际吃什么、吐什么。**
- **khazix-writer 已经做了一半,就别重造**:读 `khazix-writer/SKILL.md` 发现它天生吃任意素材、自带 HKR 选题质检、承载笔调。所以本 skill 的写作步**全权委派**给它,选题步只做轻量内联——薄编排层不重写下游能力。**复用的前提是先知道下游到底覆盖了多少。**
- **机械环节真验、编辑环节留人**:`library export --account "刘备教授"` 对真实库跑通(count 10、`contentPath` 真实存在)——选料命令确认无误。但「选题→出稿」是编辑活、产出安哥的真实笔调内容、且 skill 本就要求两个人工检查点,所以**端到端样例必须主会话与安哥一起跑**(他在检查点 1 拍板选题),不擅自生成整篇。M15 据此诚实标「skill 已交付,端到端样例待跑」,不假装验收完成。
- **边界写进 skill 本体**:`wx-kit-compose` 只经 wx-kit 的 CLI/导出文件取数据、绝不 import 应用代码——换任何下载器/agent 都按同一契约接入。这把 v0.4.0「wx-kit 只供料」的边界从 PRD 落到了 skill 的硬约束里。

> 一句话：M15 的方法论是**「按交付物形态选执行法、编排前先把被引用的 skill 读穿(别信 description 的顺口话)、下游能覆盖的就别重造、机械环节真验编辑环节留人在环中」**——v0.4.0 至此收口:存储夯实、文库供料、创作闭环——端到端样例已与安哥实跑通一次（刘备教授 3 篇市场日评 → 选题 A「Token 从狂烧到精算」→ khazix-writer 初稿，两个检查点都如约停下等他拍板），三里程碑全部落地。

## 27. v0.4.0 发版：被一句陈旧的「待发版」绊了一跤（2026-06-23）

M13–M15 合齐后给 v0.4.0 出包发版。发版流程本身按规约六步顺顺当当跑完，真正值得记的是发版前那一跤——它和 §22、§26 是同一个病根的第三次复发。

- **「文档里的旧话」被我当事实重复了一整个会话**：我好几次跟安哥说「v0.3.0 功能完成待发版」，因为 ROADMAP 的「下一步」段一直这么写。直到安哥一句「v0.3.0 早就发版了」，我去查 `git tag` + `gh release list`，30 秒就真相大白——v0.3.0 在 2026-06-16 就发了 tag + Release，那句「待发版」从发版那天起就 stale 了，是 v0.3.0 发版时漏刷「下一步」段留下的债，我又原样背了一路。**这和 fakeid（spec 说有、代码没有）、docs/plans 路径（信 skill 默认没核项目约定）是同一个错：把文档/默认值当事实，没回源验证。** 教训钉死成一条操作习惯：**凡涉及「发没发版/什么状态」，先 `git tag` + `gh release list` 看实际，别信 ROADMAP 的散文。** 而且发版必须当场刷新所有状态文档（README + ROADMAP 当前状态/下一步 + 迭代段标题），否则下一个版本又踩同一处——这正是 memory「发版要刷 README」该扩到「连 ROADMAP 下一步一起刷」。
- **发版规约六步，一步没省**：① `package.json` + lock 只 sed 改 version 行（不跑 `npm version` 免重排配置）；② 写 `docs/releases/v0.4.0.md`；③ 国内镜像出 mac dmg(arm64+x64) + win nsis；④ **真机验证打包后的 .app**——`auth-status` exit 0 证 undici external 站得住，外加专门验了 M14 新命令 `library export` 在打包二进制里也跑通（count 10），新版有新 CLI 就顺手验新 CLI；⑤ README 版本位全刷（徽章/安装包名/已落地段/项目状态 changelog）；⑥ commit + annotated tag + 推送。
- **防 Draft 残留：先建空 Release 再逐个传包**：`gh release create` 是「建草稿→传附件→最后 publish」，中途被杀会留个外部不可见的 Draft。三个包共约 380MB，所以先 `gh release create`（不带附件，秒 publish），再 `gh release upload` 逐个传——任何一个传断都不影响 Release 已发布的事实，重传那个即可。全程 unset 代理直连（8118 代理传 github 大文件会卡）。

> 一句话：v0.4.0 发版的方法论是**「状态从 git tag/gh release 回源核实、别信文档旧话；发版当场刷全部状态文档不留债；规约六步一步不省、新版顺手验新 CLI；先发布空 Release 再逐个传包防 Draft 残留」**——技术动作不难，难的是不让一句没人维护的散文把你带沟里。

## 28. v0.5.0 CLI 体验：分流真 bug、文档 vs 实践、终评抓住 per-task 漏网（2026-06-28）

v0.5.0 把 CLI 从「能被 agent 调」打磨到「顺手、自洽、与 GUI 同库」（M16 分流+help/version、M17 补命令+同库+抽订阅检查、M18 首启建 PATH 软链）。三个里程碑全程 subagent 驱动开发，方法论上有几处值得钉死：

- **需求落点：先被纠「PRD 呢」**。我一上来用「第一性原理不为惯例造文档」把 PRD 砍了、只写 spec，被安哥当场质疑「后面怎么验收」。这是 §22/§26/§27 同一病根的又一次复发——**把「省文档」当聪明，却省掉了承载验收契约的东西**。本项目每版必出 `docs/PRD-vX.Y.Z.md`，§4 逐条可勾验收是验收依据，spec（怎么做）替代不了。已固化成 memory。
- **分流是个真 bug，不是「加 flag」**：`-h/--help/-v/--version` 是 flag 不是子命令，而 `main.ts` 的 `isCliInvocation` 只认子命令 → `wx-kit --version` 会误进 GUI。修法不是堆 commander 选项，而是把分流判定抽成纯函数 `electron/cli-dispatch.ts`（可单测）让 flag 也进 CLI。**「help/version 该用哪种形式」安哥问我，我给的真实判断是「别二选一」**——flag 和子命令 commander 几乎都白送，全行业肌肉记忆是 `--help`，强行只留子命令是反直觉；`--version` 出裸 semver（agent 也好 parse），help/version 走 stdout、报错 usage 走 stderr。
- **文档 vs 实践：先改文档**：M16 评审按 AGENTS.md「注释用英文」flag 了中文注释，但全库注释从第一天就是中文——强约束与实践 100% 相反。按安哥「先改文档再改实践」，改 `AGENTS.md:48`（注释中文为主，只标识符/commit/PR 英文）、本轮发现作废、固化 memory，后续 reviewer 的 global-constraints 不再带这条。**约束与既成事实冲突时，回源核实哪个是真权威，别让一条没人执行的规则一路 flag 下去。**
- **就地改善边界**：M17 把 `runSubscriptionCheck` 从 `ipc.ts` 闭包抽到 `electron/services/subscription-check.ts`，依赖全注入（subs/session/downloadRefs/log/onEmit），GUI 与 CLI 共用。关键是**行为保真**——ipc 包装层保留 `if (result.note !== 'no-accounts') subsAuthExpired = result.authExpired` 精确复刻原语义（原 no-accounts 路径不碰该标志）。
- **终评抓住 per-task 漏网的 I1**：14 个任务逐任务评审全绿，但**整分支终评（opus）发现一个 cross-cutting 缺陷**——CLI `subscription check-now` 的日志只写 stderr、没落盘到 GUI 可观测性读的 check log，download 模式下 `downloadRefs` 忽略 `source`、不记下载历史，与验收「与 GUI 一致的日志落盘」相悖。**per-task 评审只看单 diff，看不到「CLI 行为是否和 GUI 一致」这种跨文件契约**；这正是终评的价值。修了（而非弱化验收）：CLI 落盘 `subscriptions-check.log` + `subs.appendCheckLog` + 记 history。
- **subagent 模型分层省成本**：纯转录任务（计划里有完整代码）派 haiku，集成/改既有文件派 sonnet，终评派 opus；reviewer 按 diff 大小给模型。一次 reviewer 撞会话限额，重派即恢复（diff 包是文件、不在主上下文，重派零成本）。
- **M18 真 e2e 验证**：软链是「点弹窗→建 `~/bin/wx-kit`→写 `~/.zshrc`」，bash 点不了 Antd 弹窗，写了个一次性 Playwright 驱动（隔离临时 HOME 防污染开发机），真实点「创 建」后断言 symlink 指向 execPath、`.zshrc` 含 PATH 行、重启不再弹。**纯逻辑 TDD 覆盖（Task 2 对真 temp dir）+ 关键交互真 e2e，比「靠 review 脑补」实在。**
- **合并后被安哥一句「怎么弹钥匙串、GUI 自己关了」抓出真回归**：我合并前**漏跑了完整 GUI 回归 `npm run test:e2e`**（只跑单测 + 那个隔离 HOME 的软链脚本），还把「软链脚本过了」说成「e2e 验证过」——拿子集冒充全量。补跑全量时才暴露：`gui.e2e.mjs` 只隔离 `--user-data-dir`、**不覆盖 `HOME`**，于是 M18 首启 Modal 在**真实 home** 弹出、被 e2e 交互误触发，**在安哥真机上瞬时建了 `~/bin/wx-kit` 软链**（随后被清、`~/.zshrc` 没被改，无残留但确实碰了机器）；「钥匙串」是 Electron 用全新 profile 访问 macOS Keychain（safeStorage）的系统提示，「GUI 自己关」是脚本 `app.close()`。修法：e2e seed 加 `cliLinkPrompted: true` 让首启 Modal 在自动化里**永不触发**，复跑确认全过且跑前跑后真机 `~/bin`/`~/.zshrc` 零变动。**教训：①「验证过」必须指明跑了哪条，别拿子集冒充全量；② e2e 跑在真实 HOME，任何「写 HOME 外 / 动系统」的功能默认在 e2e seed 关掉，手动验真 Electron 用临时 HOME。**（已固化 memory）

> 一句话：v0.5.0 的方法论是**「每版必出 PRD（验收契约别省）；分流类问题抽纯函数可单测；约束撞实践先回源改文档；重构靠注入依赖保行为；逐任务评审之外必做整分支终评抓 cross-cutting；模型按任务分层；关键交互写真 e2e 别脑补；『验证过』要指明跑了哪条、e2e 跑在真实 HOME 别污染开发机」**。
