# wx-kit v0.5.4 产品需求文档（迭代 PRD）

> 本文件是 **v0.5.4 迭代**的需求源头与验收依据。订阅检查专项：防重复检查、失败可观测、请求瘦身。
> 实现计划见 `docs/plans/2026-07-16-m22-subscription-check-hardening.md`；状态/进度见 `ROADMAP.md`。

## 1. 一句话定义

把订阅自动检查从「同一时段会重复跑、失败只有个数、每号固定翻 4 页」修成「一次只跑一个、失败可点开看明细、平时每号 1 次请求且空窗后不漏」。

## 2. 背景（2026-07-15 安哥真机日志实证）

- **同一时段两条记录**：调度器每 60s tick,判定依据 `lastRunAt` 只在检查结束时写,且无「检查进行中」守卫;检查串行+随机延迟+分页,账号多/网络慢时耗时超过 1 分钟 → 下一个 tick 又启动一个并发检查。真机证据:07-13 两条结果完全相同(`new=10`,同批文章数了两遍);06-18 曾 4 秒内 9 条。
- **「失败 x」不可回答**：`check-subscriptions` 明明带回了每号错误(`r.error`),`runSubscriptionCheck` 汇总时丢弃,日志只剩 `failed=N`——失败是频控、登录失效还是网络,无从知晓。
- **每号 4 次请求**：检查取「最近 20 篇」,而微信 appmsg 接口每页实测只回 ~5 篇 → 翻 4 页凑满。安哥提出砍到 5(订阅号最多日更 1 篇);但检查只在应用开着时发生(真机有 9-10 天空窗),固定 5 会在空窗后**静默永久漏文章**(水位跳到最新)。定案:去掉固定 count,改「翻到水位为止」。

## 3. 功能需求

### R1 · 检查防重入（里程碑 M22）

- 调度器 tick 加 in-flight 守卫:上一次触发的检查还没结束,本 tick 跳过(不排队、不并发)。
- ipc 层 `runSubscriptionCheck` 加共享 in-flight:自动检查与手动「检查更新」重叠时并入同一次运行,不重复请求、不重复计数。
- 语义不变:`lastRunAt` 仍在检查结束时写(「跑完才算跑过」)。

### R2 · 失败明细可观测（里程碑 M22）

- `CheckLogEntry` 增加 `failures?: { nickname, error }[]`(每号失败原因,错误话术沿用 core 已归一的中文,如「微信频率限制（200013）」)。
- 订阅页检查记录里「失败 x」(x>0 且有明细时)变为可点击,弹窗列出**哪个号、什么原因**;旧记录(无明细字段)保持纯文本。
- 落盘日志行(`subscriptions-check.log`)与 CLI `subscription check-now` 的 JSON 输出同步携带失败明细。

### R3 · 检查请求瘦身：翻到水位为止（里程碑 M22）

- 新增 `listArticlesSince`(mp-client):从最新往回翻页,**整页都比水位新才继续翻**,看见已读文章即止;封顶 20 篇(保留原防漏上限)。
- 订阅检查改用它:平时(日常打开应用)每号恒 **1 次请求**(原 4 次,省 75%);空窗多日后自动翻深,不漏文章。
- 检查语义不变:水位判定、失败隔离、频控不重试、账号乱序与随机间隔全部保持。

**存储影响**：`subscriptions.json` 的 checkLog 条目新增可选字段 `failures`,向后兼容(旧条目无此字段,读取不受影响)。

### R4 · 发版（v0.5.4）

按发版规约走完整发版。

## 6. 非目标

- **跨进程防重入**(GUI 自动检查与 CLI `check-now` 同时跑)——两进程场景罕见且互不损坏数据,不加文件锁复杂度。
- **失败自动重试/退避**——维持「频控不重试」原则(devlog §21-23),失败等下一轮。
- **checkLog 结构升级/迁移**——只加可选字段,不动版本号。
