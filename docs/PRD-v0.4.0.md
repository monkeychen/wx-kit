# wx-kit v0.4.0 产品需求文档（迭代 PRD）

> 本文件是 **v0.4.0 迭代**的需求源头与验收依据，建立在 `docs/PRD.md`（v0.1.0）、`docs/PRD-v0.2.0.md`、`docs/PRD-v0.3.0.md` 之上。
> 「怎么做」见设计 spec `docs/superpowers/specs/2026-06-22-v0.4.0-agent-feed-and-storage-design.md`；
> 逐里程碑实现计划见 `docs/plans/`；状态/进度见 `ROADMAP.md`。
> 刻意写精不写厚——本迭代真正的增量是 **§4 验收标准**（逐条可勾）。

## 1. 一句话定义

把文库从「自己看的归档」升级为**可被 AI agent 消费的素材源**——选中文库文章（或某日自动订阅下载的文章）一键导出为结构化素材，交由 wx-kit 之外的 Claude Code skill 驱动「选题 → 写作 → 审阅」的人在环中创作流水线；并在此之前**夯实文件存储地基**（原子写 + 写串行化 + 损坏可重建）。

## 2. 目标与非目标

### 2.1 目标
- **存储地基**：消除「崩溃/断电损坏整个索引」「后台订阅下载与前台操作并发丢更新」两个真实隐患，并提供损坏后的一键重建。
- **供料能力**：提供选料导出（CLI + GUI），把选中的文章变成 agent 能直接吃的结构化清单（JSON + content.md 路径）。
- **闭环验证**：交付一条版本化在仓库 `agent/` 的样例 skill，串起「文库选料 → 选题候选 → 人工拍板 → 初稿 → 人工审定」，证明「文库 → 成稿」闭环成立。

### 2.2 非目标（v0.4.0 明确不做）
- **wx-kit 内置创作 GUI 模块**——编排活在应用之外的 skill 里，应用只供料（已定边界，勿回退）。
- **全自动端到端生成**——样例流水线强制人在环中（检查点）；不做无人工介入的一气呵成。
- **审阅段高级校验**（事实核查、查重/原创度检测）——本迭代审阅段为可选、轻量。
- **SQLite / 数据库化**——存储加固已足够覆盖当前规模与并发，不上数据库（沿用既定「无数据库」决策）。
- **素材的图片打包**——agent 读正文文本即可，图片路径已内联在 content.md。
- **洗稿 / 自动转发**——见 §6 立场。

## 3. 功能需求

### R1 · 存储加固（里程碑 M13）

设计依据见 spec「M13」节。

**需求**：
- **原子写**：`library.json` / `history.json` / `subscriptions.json` 的写入改为「写临时文件 → 原子 `rename` 替换」，进程中途中断不再损坏索引。
- **写串行化**：同一索引文件的「读-改-写」整体串行（按文件绝对路径加模块级异步锁），杜绝后台订阅自动下载与前台操作并发时的丢更新。
- **损坏可重建**：新增「重建索引」能力——扫描库根各文章目录的 `meta.json` 重建 `library.json`；CLI `library rebuild` + GUI 在文库加载失败时提供「重建索引」入口；索引损坏的错误提示从「手删重来」改为「可重建修复」。

**存储影响**：无新增持久化文件/字段；改变的是写入方式（原子）与并发保护。`exports/`（见 R2）与非文章目录在重建时被忽略。

### R2 · 供料能力（里程碑 M14）

设计依据见 spec「M14」节。

**需求**：
- **`library export` CLI**：输出选中文章的 JSON 清单（stdout 纯 JSON，守 PRD §F4 CLI 契约），每篇含 `{id, title, account, author, publishTime, sourceUrl, dir, contentPath}`；**不内联正文**，给 content.md 绝对路径供 agent 按需读。
- **选料三选器，可组合**（交集语义）：
  - `--ids a,b,c`（按文章 id 显式选）；
  - `--since YYYY-MM-DD`（按 `downloadTime` 过滤——「某日及之后自动订阅下载的」用这个，不引入新水位状态）；
  - `--account <昵称>`（按公众号名选，大小写不敏感包含匹配；`ArticleMeta` 无 fakeid，故仅按昵称）。
  - 无选择器时报错避免误导全库；全库须显式 `--all`。
- **GUI「导出选中为素材」**：复用 M9 文库多选，选中 ≥1 篇后工具栏出现该按钮；点击把素材清单写到库内 `exports/<时间戳>.json` 并提示路径（+「在文件夹显示」）。外部 skill 接此文件即可。

**存储影响**：库根下新增 `exports/` 目录（GUI 导出的素材清单落此），与 `library.json` 等同级；不进文库索引、重建时忽略。

### R3 · 贯通样例 skill（里程碑 M15）

设计依据见 spec「M15」节。

**需求**：
- 交付一个 **Claude Code skill**（非 app 代码），版本化在**仓库 `agent/` 目录**，可经 skill-kit 安装；与 app 代码物理隔离，只经 wx-kit 的 CLI / 导出文件消费数据，不 import app 代码或直碰索引文件。
- 串起带检查点的端到端流程（人在环中）：
  1. **选料**（调 `library export` 或读 GUI 导出的 `exports/*.json` → 读各 content.md）；
  2. **选题候选**（以 `hv-analysis` 思路从素材提炼若干选题方向，带角度/理由）；
  3. **【检查点：用户拍板选题】**；
  4. **写作初稿**（选定选题 + 素材喂 `khazix-writer`）；
  5. **【检查点：用户审定】**。
- **至少打通选料 + 选题 + 写作**；审阅/校验作为可选第三段，不强求一版到位。
- 配套集成文档：CLI 契约速查 + skill 安装/使用 + 打包后 CLI 路径（mac/win，引用 AGENTS.md 陷阱）。

## 4. 验收标准

### R1 / M13 · 存储加固
- [ ] 三个索引（library/history/subscriptions）写入均走原子写；注入「写到一半失败」后原文件内容完好、不损坏。
- [ ] 两个独立 `Library` 实例并发 `add` 不丢更新（最终索引含两者）；同一路径写串行、不同路径可并行（单测断言执行序）。
- [ ] `library rebuild`（CLI）扫描文章目录 `meta.json` 重建 `library.json`：含 N 个有效 meta 的目录树重建出 N 篇；无 meta 目录与 `exports/` 被跳过；输出重建结果 JSON（扫描/重建/跳过数）。
- [ ] GUI 在文库索引损坏/加载失败时提供「重建索引」入口并可成功重建；损坏错误提示指向重建而非手删。
- [ ] `npm test` / `tsc` / `lint` 全绿；并发与原子写用例覆盖。

### R2 / M14 · 供料能力
- [ ] `library export --ids a,b,c` 输出含这些文章的 JSON 清单，字段含 `contentPath`（content.md 绝对路径）且路径真实存在。
- [ ] `library export --since YYYY-MM-DD` 只含 `downloadTime` ≥ 该日 00:00 的文章；`--account <昵称>` 只含该号文章（昵称包含匹配）；多选器组合取交集。
- [ ] 无选择器时报错、不导出全库；`--all` 才导出全库。stdout 为纯 JSON（守 CLI 契约，退出码 0/1/2）。
- [ ] GUI 文库多选 ≥1 篇后出现「导出为素材」按钮；点击在库内 `exports/<时间戳>.json` 生成清单、提示路径、可「在文件夹显示」；清单内容与 CLI 同源一致。
- [ ] `exports/` 不被文库扫描/重建当作文章目录。
- [ ] 选料过滤与清单组装为纯逻辑、TDD 覆盖；CLI/IPC 薄委派；`npm test` / `tsc` / `lint` 全绿。

### R3 / M15 · 贯通样例 skill
- [ ] 仓库 `agent/` 下有样例 skill，可经 skill-kit 安装；不 import app 代码、不直读 `library.json`，只经 CLI/导出文件取数据。
- [ ] 用真实文库素材跑通一条样例（主会话/本地）：选 2–3 篇 → 出选题候选 → 用户选定一个 → 出一篇初稿，两个检查点均停下等用户。
- [ ] 至少覆盖选料 + 选题 + 写作三步；复用 `hv-analysis`/`khazix-writer`，wx-kit 内无新增创作模块。
- [ ] 配套集成文档可照做跑通（CLI 契约 + 安装使用 + 打包后 CLI 路径）。

## 5. 里程碑拆分

| 里程碑 | 范围 | 状态 |
|--------|------|------|
| **M13** | 存储加固：原子写 + 写串行化 + rebuild 恢复（R1） | ⏳ 待实现 |
| **M14** | 供料能力：`library export` CLI + GUI「导出选中为素材」（R2） | ⏳ 待实现 |
| **M15** | 贯通样例 skill（仓库 `agent/`）：选料 → 选题 → 写作，带检查点（R3） | ⏳ 待实现 |

**顺序**：M13 先做（地基，后两者写盘都受益于原子写/写锁）→ M14（供料，依赖稳定存储）→ M15（样例，消费 M14 的导出）。三者强先后。

## 6. 立场：素材用途（强约束）

- 文库素材用于**选题 / 研究 / 启发 + 人工主导的创作**，**不做洗稿、不做自动转发或批量改写发布**。
- 样例创作流水线**强制人在环中**（选题、定稿两个检查点不可绕过）——AI 化的是收集素材、出初稿、一致性检查这类重复劳动，**品味与定稿权留在用户手里**。
- 这条是产品定位的一部分，约束 M15 样例 skill 的设计，不只是免责声明。
