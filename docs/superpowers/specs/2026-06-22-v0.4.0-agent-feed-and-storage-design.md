# v0.4.0 设计稿：文库供料 agent + 存储加固

> 本文是 v0.4.0 的「怎么做」设计依据，配合需求源头 `docs/PRD-v0.4.0.md`。
> 经 2026-06-22 与安哥 brainstorming 敲定的关键决策见「决策记录」。

## 决策记录（brainstorming 结论）

1. **范围**：主线是「文库供料 agent 创作流水线」；存储加固（原子写 + 写锁 + rebuild）作为**前置地基**一并做。
2. **架构边界**：**wx-kit 只供料，不内置创作模块**。选题/写作/审阅的编排活在 wx-kit 之外的 Claude Code skill 里，复用现有 `hv-analysis`/`khazix-writer`，**人在环中**（带检查点，非全自动）。
3. **交付边界**：供料侧（CLI + 存储加固）+ 一条贯通的样例 skill（至少打通选题 + 写作），证明闭环成立。
4. **导出形式**：`library export` 输出 **JSON 清单 + content.md 绝对路径**（不内联正文），agent 按需读盘。
5. **GUI 入口**：文库加「导出选中为素材」按钮（复用 M9 多选），写素材清单到库内 `exports/` 目录。
6. **样例 skill 交付位置**：版本化进**仓库 `agent/` 目录**，与 app 代码物理隔离；不进全局 `~/.claude/skills`。
7. **版权立场**：素材用于选题/研究/启发 + 人工主导创作，**不做洗稿/自动转发**；样例流水线强制人在环中。

里程碑：接 v0.3.0 的 M12，顺延 **M13 存储加固 → M14 供料能力 → M15 样例 skill**。

## M13 · 存储加固（地基）

### 问题（现状）
`Library`/`History`/`Subscriptions` 均「整文件 `JSON.parse` 读出 → 改内存 → `writeFile` 整文件覆盖」，且无写串行化：

- **非原子写**：`writeFile` 先截断再写，进程中途被杀/断电会留下半截 JSON。下次 `read()` 抛 "index is corrupt — delete it to reset"，**整个索引报废**（磁盘文章文件仍在，但应用内全不可见）。这是唯一会真丢数据的路径。
- **读-改-写竞态**：每次写中间有 `await`，Electron 主进程单线程也会让两个异步操作交错。后台订阅自动下载（`recordHistory` + `library.add`）与前台用户操作（`remove`/`markDeleted`）并发时，各自读旧文件 → 改 → 写，**后写覆盖先写，一条更新静默丢失**。已开自动订阅下载，此并发是现实。

### 方案

1. **原子写工具**（核心层新增，如 `src/core/atomic-write.ts`）：
   - `atomicWriteFile(path, data)`：写到同目录临时文件 `path + '.tmp-' + rand` → `fsync` → `rename(tmp, path)`。同一文件系统 `rename` 原子，替换不留半截。
   - `Library`/`History`/`Subscriptions` 三处 `write()` 统一改走它。
2. **按路径 keyed 的异步写锁**（核心层新增，如 `src/core/path-lock.ts`）：
   - 模块级 `Map<absPath, Promise>` 串行队列：`withPathLock(absPath, fn)` 让同一文件的读-改-写整体串行。**必须模块级**——因为每个 IPC handler 都 `new Library(root)`，实例级锁挡不住跨实例并发。
   - 三个索引类的「读-改-写」公开方法（`add`/`remove`/`append`/`markDeleted`/`updateWatermark` 等一切会写的）包进 `withPathLock(indexPath, …)`。
   - 纯读方法（`list`/`get`/`search`）不加锁（读无损）。
3. **rebuild 恢复**：
   - 核心层 `rebuildLibrary(root)`：扫 `root` 下各文章目录的 `meta.json`，重建 `library.json`（忽略无 meta 的目录与 `exports/` 等非文章目录）。纯逻辑可注入 fs。
   - CLI `library rebuild`（输出重建结果 JSON：扫描数/重建数/跳过数）；IPC `library:rebuild` 供 GUI 调。
   - 损坏处理升级：`read()` 检测到 corrupt 时，错误信息从「手删」改为「可运行 rebuild 修复」；GUI 侧在文库加载失败时给「重建索引」入口（最小：设置页或文库空态按钮）。

### 测试
- `atomicWriteFile`：写后内容完整；模拟写中途失败（注入抛错的 write）后原文件不被破坏。
- `path-lock`：两个并发 `withPathLock` 对同一路径串行执行（断言执行序）、不同路径可并行。
- 索引类并发：两个 `new Library(root)` 实例并发 `add` 不丢更新（断言最终 articles 含两者）。
- `rebuildLibrary`：给定含 N 个 meta.json 的目录树重建出 N 篇；跳过无 meta 目录与 `exports/`。

## M14 · 供料能力（wx-kit 侧）

### `library export` CLI
- 选料三选器，可组合（取交集语义：同时给 `--account` 和 `--since` = 该号且该日之后）：
  - `--ids a,b,c`：按文章 id 显式选。
  - `--since YYYY-MM-DD`：按 `meta.downloadTime` 过滤「该日 00:00 之后下载的」。**「今日/某日自动订阅下载的」用这个，不引入新水位状态。**
  - `--account <昵称>`：按公众号名选（大小写不敏感包含匹配）。注：`ArticleMeta` 不含 fakeid，故按号选只能用昵称，不支持 fakeid。
  - 无选择器时报错（避免误导出全库）；要全库须显式 `--all`。
- 输出（stdout 纯 JSON，守 PRD §F4 CLI 契约）：
  ```json
  {
    "ok": true,
    "count": 2,
    "articles": [
      {
        "id": "…", "title": "…", "account": "…", "author": "…",
        "publishTime": "…", "sourceUrl": "…",
        "dir": "/abs/path/to/article",
        "contentPath": "/abs/path/to/article/content.md"
      }
    ]
  }
  ```
- 只给路径不内联正文（stdout 轻），agent 拿清单按需读 `content.md`，复用磁盘已有产物。
- 纯逻辑（选料过滤 + 清单组装）抽纯函数 TDD；CLI 壳薄委派。

### GUI「导出选中为素材」
- 复用 M9 文库多选：选中 ≥1 篇后工具栏出现「导出为素材」按钮。
- 点击 → 走与 CLI 同一份组装逻辑生成清单 → 写到库内 `exports/<YYYYMMDD-HHMMSS>.json` → 反馈提示路径 + 「在文件夹显示」。
- `exports/` 目录由本功能创建；`rebuildLibrary` 与文库扫描忽略它。
- IPC `library:exportMaterial(ids)` → 返回写出的文件绝对路径。

## M15 · 贯通样例 skill（外部编排，仓库 `agent/`）

### 形态
- 一个 **Claude Code skill**（非 app 代码），放仓库 `agent/`（如 `agent/wx-kit-compose/`），随项目版本化、可经 skill-kit 安装。
- 带检查点的端到端流程（人在环中）：
  1. **选料**：调 `wx-kit` 打包后 CLI `library export`（或读 GUI 导出的 `exports/*.json`）拿清单 → 读各 `content.md`。
  2. **选题候选**：以 `hv-analysis` 的思路，从素材里提炼若干选题方向（带角度/理由）。
  3. **【检查点 1：你拍板选题】**。
  4. **写作初稿**：以选定选题 + 素材喂 `khazix-writer` 出初稿。
  5. **【检查点 2：你审定】**。
- **至少打通选料 + 选题 + 写作**；审阅/校验作为可选第三段，不强求一版到位。
- 配套集成文档（`agent/README.md` 或 `docs/`）：CLI 契约速查 + skill 安装/使用 + 打包后 CLI 路径（mac/win，引用 AGENTS.md 陷阱）。

### 与 app 的边界
- skill 只经 wx-kit 的 **CLI / 导出文件**消费数据，绝不 import app 代码或直碰 `library.json`。
- wx-kit 不感知这个 skill 的存在；换任何 agent/skill 都能照同一 CLI 契约接入。

### 验证
- 用真实文库素材跑一条样例（主会话/本地）：选 2–3 篇 → 出选题候选 → 选一个 → 出一篇初稿，证明闭环。
