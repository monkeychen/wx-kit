# v0.5.0 设计 — CLI 体验优化(M16–M18)

> 状态:设计已确认,待落实现计划。
> 目标:把 CLI 从「能被 agent 调」提升到「顺手、自洽、与 GUI 同库」。三条主线:
> ① 帮助/版本可用且分流正确;② GUI 能力按「agent 用得到」补齐 CLI 入口;③ 首启引导建 PATH 软链。
> 不变量与陷阱见 `AGENTS.md`;CLI 输出契约见 `docs/PRD.md` §F4(stdout 纯 JSON、stderr 进度、退出码 0/1/2)。

## 背景与问题

CLI 与 GUI 同一二进制、带子命令即进 CLI 模式(`electron/main.ts`)。当前存在三类问题:

1. **帮助/版本拿不到**:`isCliInvocation` 只认子命令集合,`--help`/`-v`/`--version` 是 flag 不是子命令 → `wx-kit --help`、`wx-kit --version` **误进 GUI**(弹窗)。commander 内部支持 `-h`,但根本到不了 CLI 层。
2. **GUI/CLI 能力不齐**:文库关键词搜索、删除、订阅检查、设置读写等只有 GUI 入口,agent 无法脚本化。
3. **首装即用门槛**:打包后 CLI 真身在 `…/wx-kit.app/Contents/MacOS/wx-kit`,用户要么敲全路径要么自建软链,无引导。

另有一个**既有隐性不一致**:CLI 各命令 `--out` 默认硬编码 `~/Documents/wx-kit`(`src/cli/index.ts` 的 `defaultLibraryRoot`),无视用户在设置里改过的 `libraryRoot`。GUI 走 `settings.libraryRoot`、CLI 走硬编码 → 两模式默认操作不同库。本版一并消除。

## 范围(已确认)

- **补齐范围(按「agent 用得到」筛选)**:文库 `search`/`remove`、订阅 `list`/`check-now`、设置 `get`/`set`。
- **不补**:下载历史 list/remove/clear、`readContent`(export 已给 content.md 路径)、`coverName`/`chooseDir`/`reveal`/`openLog`(GUI 专属,YAGNI)。
- **软链平台**:仅 macOS + Linux(unix symlink);Windows 不做(与 ROADMAP「Windows CLI stdout 未正解」候选一致)。

## M16 — 模式分流修复 + help/version

**根因**:分流判定只识别子命令。**改法**:

- `electron/main.ts` 的分流扩展为:`argv[0]` ∈ 子命令集合 **或** `argv` 含 `-h|--help|-v|--version` **或** `argv[0] === 'help'` → 进 CLI。**无参 `wx-kit` 仍进 GUI**(首启要的就是这个,见 M18)。
- **flag 与子命令都提供**(commander 几乎零成本白送,不二选一):
  - `.version(pkg.version, '-v, --version')` → `wx-kit -v` / `wx-kit --version` 输出**裸版本号** `0.5.0\n`(全行业惯例,agent 也好 parse,不套 JSON)。
  - `-h`/`--help` 每层自动可用;顶层 `wx-kit -h` 列出全部子命令与简介。
  - `help` 子命令(commander 内建):`wx-kit help`、`wx-kit help crawl` 查单命令用法。
- **输出去向**:help/version 是用户主动查询,走 **stdout**;仅报错触发的 usage 走 stderr。拆掉现在 `configureOutput` 把 `writeOut`/`writeErr` 都打到 stderr 的写法 → `writeOut→process.stdout`、`writeErr→process.stderr`。
- 退出码:help/version 正常退出 0(`runCli` 已处理 `commander.helpDisplayed`/`commander.version`,确认仍走 0)。

**验收**:`wx-kit --version` 打 `0.5.0` 到 stdout 退 0;`wx-kit --help`/`-h` 打全量帮助到 stdout 退 0;`wx-kit help crawl` 打 crawl 用法;`wx-kit`(无参)仍开 GUI;以上均不再弹 GUI。

## M17 — CLI 补齐(文库 / 订阅 / 设置)

通则:输出纯 JSON 到 stdout;凡涉及库根的命令,`--out` **默认回落 `settings.libraryRoot`**(消除 GUI/CLI 不同库),用户显式给 `--out` 时以显式为准。CLI 在 `runCli` 内构造 `SettingsService(app.getPath('userData'), defaultDocumentsRoot)` 读取设置。

### 文库
- `library search <keyword> [--account <name>] [-o <dir>]` → `{ ok, items }`。复用 `Library.search(kw)`,`--account` 再过滤。
- `library remove --ids <csv> [-o <dir>]` → 删文件 + 对每个 id `History.markDeleted`(与 GUI 的删-史联动一致)→ `{ ok, removed }`。无 `--ids` → `{ ok:false, error:{ code:'NO_SELECTOR' } }` 退 2。

### 订阅
- `subscription list` → 复用 `subscriptions:list` 的合并逻辑(`accountsFromHistory` + `subs.list()` merge),输出 `{ ok, accounts, lastRunAt, nextCheckAt, authExpired }`。
- `subscription check-now` → 触发一次手动检查,输出 `{ ok, accounts, newFound, failed, note? }`。命中频控不重试(沿用既有约束);未登录 → `note:'no-session'`、`ok:true`(与 GUI 行为一致,不算错误)。
  - check-now headless 安全:CLI 模式 main.ts 已注册 no-op `window-all-closed`,PDF 离屏窗口可用;`emitSubsUpdated()` 在无窗口时自然 no-op。

### 设置
- `settings get [key]` → 无 key 输出全量 `AppSettings`;给 key 输出 `{ ok, key, value }`;未知 key → 退 2。
- `settings set <key> <value>` → 校验 key 在白名单且值类型/取值合法(如 `subscriptionScheduleMode` ∈ daily|interval、数字字段可解析),写入后输出回写的全量 `AppSettings`;非法 → `{ ok:false, error }` 退 2。
- 白名单 = `AppSettings` 可由用户配置的键(`libraryRoot`、`defaultFormats`、`subscriptionAutoCheck`、`subscriptionCheckTime`、`subscriptionNewArticleAction`、`subscriptionScheduleMode`、`subscriptionIntervalHours`、`historyRetentionDays`)。`listColumnWidths` 是 GUI 专属布局,不开放 CLI 写。

### 重构(就地改善边界)
当前 `runSubscriptionCheck` 困在 `electron/ipc.ts` 的闭包里,CLI 无法复用。抽到独立模块 **`electron/services/subscription-check.ts`**,签名参数化:

```
runSubscriptionCheck(deps: {
  settings, subsFor, getSession, makeMpFetch,
  onEmit?: () => void,           // GUI 传 emitSubsUpdated;CLI 不传
  log: (entry: CheckLogEntry) => Promise<void>,  // GUI 写盘+广播;CLI 写 stderr/盘
  downloadRefs, establishWatermark,
}, trigger: 'auto' | 'manual'): Promise<{ accounts; newFound; failed; note? }>
```

`ipc.ts` 与 `src/cli/index.ts` 同时调用此函数。`downloadRefs`/`establishWatermark` 一并提取或注入(二者也仅 ipc 内有,需共享)。**约束**:该模块属主进程服务层(`electron/services/`),可 import electron;**不放 `src/core/`**(core 不得碰 electron 运行时)。

**验收**:四组命令各自输出契约 JSON;`settings set libraryRoot <x>` 后 `library list`(不带 `--out`)操作的是 `<x>`;`subscription check-now` 与 GUI「立即检查」产生一致的水位推进与日志落盘。

## M18 — 首启建 PATH 软链(macOS + Linux)

新服务 **`electron/services/cli-link.ts`**:

- `status(dir): 'linked' | 'unlinked' | 'conflict'` — `dir/wx-kit` 不存在=unlinked、是指向 `process.execPath` 的 symlink=linked、存在但指别处=conflict。
- `create(dir = ~/bin)` — `mkdir -p dir` 后 `symlink(process.execPath, dir/wx-kit)`(打包后 `process.execPath` 即 `…/Contents/MacOS/wx-kit`,正是双模式入口);conflict 时需显式覆盖。
- `pathHas(dir): boolean` — `dir` 是否在 `process.env.PATH`。
- `addToProfile(dir)` — 按 `$SHELL` 选 profile(zsh→`~/.zshrc`、bash→`~/.bashrc`,兜底 `~/.profile`),**幂等**追加 `export PATH="$HOME/bin:$PATH"`(已存在该行则跳过)。

**首启交互**(GUI 主流程):

- 时机:GUI 启动且平台 ∈ {darwin, linux} 且 `settings.cliLinkPrompted !== true` 且 `status(~/bin) !== 'linked'` → 弹一次性 Modal。
- Modal 文案为目标导向、引导下一步(非「连接失败」式干报告):「为 `wx-kit` 创建命令行快捷方式?创建后可在终端直接运行 `wx-kit …` 供 AI agent 调用。」
- 接受 → `create(~/bin)`;若 `!pathHas(~/bin)` → 二段提示「`~/bin` 不在 PATH,是否写入 `~/.zshrc`?」接受则 `addToProfile`,并提示「重开终端生效」。
- **无论接受或忽略,置 `settings.cliLinkPrompted = true`**,不再每次弹。
- conflict(已链到别处)→ Modal 提示覆盖确认。

**设置页**:新增按钮「创建命令行快捷方式」,可随时重做(显示当前 `status`);此入口绕过 `cliLinkPrompted`。

**数据变更**:`AppSettings` 增 `cliLinkPrompted: boolean`(默认 `false`),走 `SettingsService` 既有 `{...defaults, ...raw}` 合并,旧设置文件自动补默认。

**Windows**:本版不做 — 不弹此 Modal、设置页按钮在 win 上隐藏/置灰。Windows 的 CLI 可达性(console 子系统入口 / `.cmd` wrapper)留作后续候选,与 ROADMAP 一致。

**验收**:macOS 全新 userData 首启弹一次 Modal;接受后 `~/bin/wx-kit` 存在且指向 `process.execPath`、`status` 返回 linked;忽略后重启不再弹;设置页按钮能重建并正确显示状态。
