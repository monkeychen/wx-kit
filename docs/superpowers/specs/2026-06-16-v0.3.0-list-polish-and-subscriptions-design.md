# v0.3.0 设计：列表视图优化 + 公众号订阅

> 设计文档（spec）。定「做什么、为什么、边界」，不写逐步实现——实现计划见 `docs/plans/`。
> 状态：设计已确认（2026-06-16，安哥）。

## 概览

v0.3.0 含两块**相互独立**的工作，拆成两个里程碑，各自一份实现计划、分别实现合并：

- **M10 — 列表视图优化**：文库「列表」视图列宽可调 + 排序移到表头。纯 renderer 展示层改动。
- **M11 — 公众号订阅**：新增「订阅」页，类 RSS 地周期检查已订阅公众号的新文章，按配置仅提示或自动下载。

M11 依赖既有抓取（`mp-crawl` / `mp-client`）与登录（`mp-auth`）管线，但不依赖 M10。

## M10 — 列表视图优化

### 目标

当前文库「列表」视图（`src/renderer/pages/Library.tsx` + `src/renderer/index.css` 的 `.lhead/.lrow`）有两处体验问题：

1. 列宽写死（`grid-template-columns: 44px minmax(0,1fr) 132px 100px 100px 172px`），「发布时间」仅 132px，时间显示被截断看不全。
2. 排序入口在工具栏（`sort-select` 下拉 + `sort-dir` 方向按钮），与「按列看数据」的直觉脱节。

### 范围

**列宽可调**
- 列宽持久化到设置：`AppSettings` 增 `listColumnWidths`（按列 key 存像素宽，缺省回退到现有默认）。
- 表头列之间加拖拽手柄；拖动实时改 grid 列宽，松手写回设置。重开 app 保持。
- 分组态（`.list.grouped`，5 列，无独立公众号列）与非分组态（6 列）各自一套宽度。

**表头排序**
- 点击「标题 / 发布时间 / 下载时间」表头按该列排序；再点同列切升降；当前排序列显 ↑/↓ 指示。
- 底层 `sortArticles`（`src/renderer/library-view.ts`，已 TDD）与 `SortKey/SortDir` 状态不变，只把触发入口从工具栏控件改到表头点击。

**卡片视图的排序入口**（已确认的取舍）
- 卡片视图没有表头、仍需排序入口。方案：列表视图用表头排序；工具栏的排序下拉+方向按钮**保留但仅在卡片视图显示**。两个入口共改同一份 `sortKey/sortDir` 状态。
- 结果：列表视图工具栏更干净，卡片视图功能不丢。

### 不做

- 列的显示/隐藏、列拖拽重排序——YAGNI，用户只提了「列宽」和「表头排序」。
- 卡片视图布局变动。

### 测试

- 列宽持久化的纯逻辑（settings 读写默认合并）走单测。
- 拖拽交互、表头排序点击由本地 e2e 断言（列宽变化、点表头后行序变化、↑↓ 指示）。

## M11 — 公众号订阅

### 目标

当前按公众号下载全靠用户手动触发。M11 提供类 RSS 的订阅：用户订阅公众号后，wx-kit 周期性检查其新文章，按配置「仅提示」或「自动下载」。

### 关键技术约束（决定设计形态）

- **已下载文章只存公众号名（`ArticleMeta.account`），不存 `fakeid`**。而重新抓取一个号必须有 `fakeid`（抓取管线主键）。
- `fakeid` 只在两处可得：① `history.json` 里「按公众号抓取」事件（`HistorySource` 的 `account` 分支带 `fakeid`+`nickname`）；② 用户搜号结果（`MpAccount.fakeid`）。
- **纯 URL 下载来的号没有 fakeid，无法轮询**，因此不进订阅列表（已确认）。

### 导航

`下载 · 订阅 · 文库 · 设置`——「订阅」插在「下载」之后（`src/renderer/layouts/MainLayout.tsx` 的 `NAV`）。

### 数据模型 — 新增 `subscriptions.json`

库根下，与 `library.json` / `history.json` 同级：

```ts
interface SubscriptionsFile {
  version: 1
  accounts: SubscribedAccount[]
}

interface SubscribedAccount {
  fakeid: string
  nickname: string
  subscribed: boolean          // 取消订阅保留记录、仅置 false
  watermark: number            // unix 秒；createTime > watermark 即「新」
  lastCheckedAt: number | null // unix ms
  newRefs: ArticleRef[]        // 已发现、待下载/忽略的新文章（含标题，UI 直接列）
}
```

- 账号来源 = `history.json` 中 account-kind 事件（fakeid+nickname）∪ 用户搜号添加。去重按 `fakeid`。
- **初始订阅态**：从历史派生的号默认 `subscribed=false`（在列表里作为「可订阅」项出现，用户自行开）；用户主动搜号添加的号默认 `subscribed=true`（主动添加即意图订阅），同时把水位设为当前最新文章。
- **watermark 语义**：订阅那一刻把水位设为该号当前最新文章的 `createTime`。只有订阅之后发布的才算「新」，不把历史存量一股脑当新文章砸出来。每次检查后，把水位推进到本次见到的最新 `createTime`。

### 设置项（`AppSettings` 新增，统一在「设置」页配）

| 字段 | 类型 | 默认 | 含义 |
|------|------|------|------|
| `subscriptionAutoCheck` | `boolean` | `false`（opt-in） | 是否开启运行期定时检查 |
| `subscriptionCheckTime` | `string` `"HH:MM"` | `"09:00"` | 每天该时刻检查所有已订阅号 |
| `subscriptionNewArticleAction` | `'notify' \| 'download'` | `'notify'` | 发现新文章后：仅提示 / 自动下载入库 |

> 自动检查默认关，避免用户没意识到就在后台跑抓取（已确认的默认）。

### Core 层（UI 无关，可测）

- `src/core/subscriptions.ts` —— 订阅存储读写 + CRUD，仿 `History` 类（ENOENT 返空、损坏报「删除以重置」）。方法：`list` / `addAccount` / `setSubscribed` / `updateWatermark` / `setNewRefs` / `clearNewRefs`。
- `src/core/check-subscriptions.ts` —— 纯编排：给定订阅账号列表 + 注入的 `listFn`（复用 `mp-client.listArticles`）+ 各自 watermark，返回每号的新 `ArticleRef[]`。**只列表、不下载**；账号间串行 + 随机延迟；命中频控复用 `mp-crawl` 同款指数退避；**单号失败隔离**，记入结果不中断其余。回调上报进度/退避，供 UI。
- `src/core/subscription-schedule.ts` —— 纯函数 `shouldCheckNow({ now, checkTime, lastCheckedAt, autoCheck })`：判定「现在是否该触发」。把调度判断从副作用剥出来单测，覆盖：当天时刻已过且当天未检查 → true；已检查过 → false；autoCheck 关 → false；启动补检（当天时刻已过、`lastCheckedAt` 不在今天）。

### 主进程

- `electron/services/subscription-scheduler.ts` —— app 运行期每分钟 tick，调 `shouldCheckNow` 判定是否触发；**app 启动时也跑一次判定**（补检当天错过的）。触发后：
  - 跑 `check-subscriptions` 得各号新 refs，写回 `subscriptions.json`（`setNewRefs` + `updateWatermark`）。
  - `action='download'`：把新 refs 喂既有抓取/下载管线（`mp-crawl` 的下载阶段 / `DownloadQueue`）下载入库，结果照常进 `history.json`。
  - `action='notify'`：仅存 newRefs + 发 IPC 事件更新角标。
- IPC（`electron/ipc.ts` 薄委派）：`subscriptions:list` / `subscriptions:add`（搜号→加，复用既有 search）/ `subscriptions:setSubscribed` / `subscriptions:checkNow`（手动触发）/ `subscriptions:downloadNew`（把某号 newRefs 走下载管线）/ `subscriptions:dismissNew`（清某号 newRefs、推进水位）。

### Renderer

- 新页 `src/renderer/pages/Subscriptions.tsx` + 路由项。
  - 列出账号，每行：号名、订阅开关、上次检查时间、「N 篇新」角标 + 「下载 / 忽略」。
  - 顶部：「添加公众号」（搜号框 → 结果 → 加入并可订阅）、「检查更新」（手动触发 `checkNow`，显示进度/退避）。
- 「设置」页（`src/renderer/pages/Settings.tsx`）加上述 3 个配置项。
- 导航「订阅」项带新文章总数角标（所有订阅号 `newRefs` 之和）。
- renderer 只经 `window.api`（`src/renderer/api.ts`）调用，不直接 import core。

### 错误处理与诚实约束（写进实现、写进文案）

- **定时只在 app 开着时生效**。app 关着时错过的检查，下次启动补检当天一次。无真后台——单进程桌面应用的物理限制，不藏着掖着；订阅页文案明确告知。
- 检查需有效登录 session。**过期不静默失败**：订阅页显示登录引导（复用 `LoginGate`），调度暂停并在页面明示「订阅检查需重新登录」。
- 单号检查失败隔离，不连累其他号；命中频控复用退避横幅（「约 N 秒后重试 · 第 k 次」）。
- 自动下载失败：照既有 `explainError` 归一话术，结果进 history，订阅页可见。

### 测试

- Core 全 vitest TDD：`subscriptions` CRUD（含 ENOENT/损坏）、`check-subscriptions`（新检测、水位推进、单号失败隔离、退避）、`subscription-schedule.shouldCheckNow`（各时间分支 + 启动补检）。
- 主进程 scheduler 的副作用部分注入依赖（时钟、check 函数）+ 端到端验证。
- e2e（本地）：订阅页渲染、添加/切换订阅、手动「检查更新」（mock 网络），断言新文章角标与下载/忽略动作。

### 不做（v0.3.0 非目标）

- 真后台/常驻进程/系统级定时（app 必须开着）。
- 每号独立的检查时刻或独立格式——全局统一配置即可（YAGNI，可后续单议）。
- 桌面系统通知（角标够用；要不要弹系统通知后续再议）。
- 订阅导入/导出（OPML 等）。

## 里程碑与顺序

| 里程碑 | 范围 | 依赖 |
|--------|------|------|
| **M10** | 列表视图优化（列宽可调 + 表头排序） | 无 |
