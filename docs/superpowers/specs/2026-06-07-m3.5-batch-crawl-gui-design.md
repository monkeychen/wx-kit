# M3.5 设计：公众号批量爬取 GUI 页

> 状态：已与安哥对齐（2026-06-07，经可视化伴侣逐屏确认三大交互）。
> 本文件是**设计依据**；实现步骤由 writing-plans 产出到 `docs/plans/`。
> 前置：M3（CLI 切片）已合入 main——`mp-auth`/`mp-fetch`/`mp-client`/`mp-crawl` 均已验证可用。

## 1. 目标与范围

把 M3 已验证的「扫码登录 + 公众号批量爬取」能力接到 GUI，给非命令行用户一个丝滑的批量页。
**本质是 GUI 接线，核心爬取逻辑全复用 M3**，唯二要碰核心层的是给「取消」和「逐篇实时结果」开两个小口子。

**三大交互决策（可视化伴侣逐屏确认）：**
1. **单页渐进式**——一页向下生长，每步完成才显露下一步（非分步向导）。
2. **整页登录引导**——未登录时整页只显示扫码引导，不摆点不动的控件；登录后整页变流程。
3. **实时逐篇列表**——整体进度 + 预计剩余 + 实时滚动的逐篇状态（成功/已存在/失败，失败可单篇重试），当前篇高亮。

## 2. 页面与组件（renderer）

- **新导航项「批量」**：`MainLayout` 增一项（朱砂下划线选中态，与现有 下载/书架/设置 同构）。路由 `/batch`。
- **`pages/BatchCrawl.tsx`**——单页，按 `phase` 状态渐进渲染：
  - `auth` 未登录 → `<LoginGate>`（整页居中：朱砂图标 + "先登录公众号后台" + 说明 + 「扫码登录」按钮）。
  - 已登录后依次显露：
    - 搜索框（输入公众号名 → `mpSearch`）
    - 候选列表（`mpSearch` 返回的 `MpAccount[]`，每项 nickname/signature，点选其一；空结果给引导）
    - 范围 + 格式（最近 N 篇 / 日期段二选一；格式复用 `FormatPicker`）
    - 「开始爬取」按钮
    - 运行中 → `<CrawlProgress>`（实时逐篇）
- **`components/LoginGate.tsx`**：整页登录引导，点按钮调 `mpLogin()`，成功后回调刷新 auth。
- **`components/CrawlProgress.tsx`**：整体进度条 + `6/10` + 预计剩余 + 取消按钮 + 逐篇行列表（✓/⊘/✗/⟳/等待，失败行带重试）。
- **复用**：`FormatPicker`、下载页状态徽章样式（badge-ok/skip/fail）、整套设计系统 CSS（暖色编辑杂志风）。

## 3. IPC 桥（preload + ipc.ts，薄委派到 M3 服务）

新增 `window.api` 方法（`src/renderer/api.ts` 声明 + `preload.ts` 接线 + `ipc.ts` 处理）：

| api 方法 | IPC 通道 | 委派 |
|---|---|---|
| `mpLogin(): Promise<{ok:boolean; error?}>` | `mp:login` | `login()`（开扫码窗，CANCELLED 归一为 ok:false） |
| `mpAuthStatus(): Promise<{valid:boolean}>` | `mp:authStatus` | getSession + 真探测（复用 CLI auth-status 逻辑） |
| `mpSearch(name): Promise<MpAccount[]>` | `mp:search` | `searchAccount(makeMpFetch(session), token, name)`；无 session 抛 AUTH_REQUIRED |
| `mpCrawl(fakeid, range, formats): Promise<CrawlSummary>` | `mp:crawl` | `crawlAccount(...)`，带取消标志与进度回调 |
| `onCrawlProgress(cb): () => void` | `mp:crawl:progress`（主→渲染） | 见 §4 事件 |
| `mpCancelCrawl(): void` | `mp:crawl:cancel` | 置取消标志 |

- **重试失败篇**：直接复用现有 `download([url], formats)` —— 不造新接口。渲染层把该行重置为"下载中"，调 download，按结果翻 ✓/✗。
- 取消标志：ipc 层持一个模块级 `let cancelRequested`，`mp:crawl` 开始时清零，`mp:crawl:cancel` 置真，`crawlAccount` 经注入的 `shouldContinue` 读取。

### 3.1 Session 存储位置（安哥确认 2026-06-07）

- session（token+cookies）保持 M3 现状：存 **userData**（`~/Library/Application Support/wx-kit/mp-session.json`），**本地、不被 iCloud 同步**，凭证更安全。**不迁到库根**。
- 文章库根（`settings.libraryRoot`，默认 `~/Documents/wx-kit`）保持可配置、可见——两者职责分离：库根放"想看想带走的内容"，userData 放"本地凭证/应用态"。
- **关键推论**：session 一旦落盘，过期前无需人工扫码，crawl/search 全可自动化。只有 `200040`（失效）才弹回登录引导让用户重扫。

## 4. 数据流（实时逐篇的关键）

`onCrawlProgress` 推三类事件（discriminated union）：
- `{ kind: 'listed', items: { title: string }[] }` —— 列表阶段拿到 `ArticleRef[]` 后**整批发**，渲染层立刻铺出全部行（标题 + "等待"）。
- `{ kind: 'item', index: number, status: 'downloading' | 'ok' | 'skipped' | 'failed', error?: string }` —— 每篇开始/结束时发，对应行翻状态。
- `{ kind: 'done', summary: CrawlSummary }` —— 收尾。

- **预计剩余**：渲染层纯前端算——记 `start`，`已完成数 / 已耗时 → 单篇均耗 × 剩余篇`。无需后端。
- **取消**：点取消 → `mpCancelCrawl()` → 标志置真 → `DownloadQueue` 在下一篇前检查、停止、返回部分汇总（已下载保留）。

## 5. 核心层改动（唯二，纯逻辑、可单测）

- **`src/core/download-queue.ts`**：`run()` 接受可选 `shouldContinue?: () => boolean`，每篇循环顶检查；返回 false 则停止并返回已完成的部分汇总。URL 下载页不传 → 行为不变。
- **`src/core/mp-crawl.ts`**：`crawlAccount` 的 `CrawlDeps` 增 `shouldContinue?` 与 `onItem?`/`onListed?` 回调（注入）；`listArticles` 拿到 refs 后先 `onListed(refs)`，下载阶段把每篇结果经 `onItem` 上报，并把 `shouldContinue` 透传给 `DownloadQueue`。

> 取消触发"部分完成"，与去重续传天然兼容：重跑自动跳过已下。

## 6. 错误处理（反馈引导行动）

- **未登录 / 中途失效（AUTH_REQUIRED）**：任何 mp 调用抛此码 → 整页弹回 `<LoginGate>`，不裸报错。
- **搜号空结果**：提示"没找到这个公众号，换个名字试试"。**多候选**：列出让用户点选（不自动选）。
- **频控（RATE_LIMITED）**：顶部提示"微信限流，已自动放慢，可稍后重试"；底层退避已在 M3 实现。
- **单篇失败**：行内 ✗ + 错误摘要 + 重试按钮，不中断整体。
- **取消**：明确反馈"已停止，已下载的 N 篇已保留"。

## 7. 测试策略

- **纯逻辑单测**（vitest）：
  - `DownloadQueue` 的 `shouldContinue` 取消：注入第 2 篇前返回 false，断言只下了 1 篇、summary 为部分。
  - `mp-crawl` 的上报：注入假 `downloadOne` + 假回调，断言 `onListed` 先发全量、`onItem` 按序发 ok/skipped/failed、取消生效。
  - 预计剩余 ETA 计算抽成纯函数 `estimateRemaining(completed, total, elapsedMs)` 单测。
- **真链路半自动**：session 持久在 userData（§3.1），**过期前可自动化真实验证**——主会话本地跑一次"搜号 → 爬 2–3 篇 → 看进度/取消/重试"的脚本，无需每次人工扫码；只有 session 过期（`200040`）才喊安哥重扫一次。这把"全靠手动"降级为"自动验、过期才人工"。
- **GUI e2e**：Playwright Electron 端到端覆盖到"登录态存在时"的批量页渲染（用已持久化的 session 或注入假 session），断言登录后流程、逐篇列表、取消按钮存在；扫码登录窗本身不自动化（同既有约束）。
- 登录态分支、逐篇渲染、ETA 等关键逻辑抽成可测纯单元。

## 8. 非目标（YAGNI）

- **暂停/恢复**：用「取消 + 续传」替代（取消后重跑自动跳过已下），不做独立暂停态机。
- **多公众号并行**：维持单号串行（风控约束）。
- **GUI 内管理多个登录账号**：维持单 session。
