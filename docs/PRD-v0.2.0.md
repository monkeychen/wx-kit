# wx-kit v0.2.0 产品需求文档（迭代 PRD）

> 本文件是 **v0.2.0 迭代**的需求源头，建立在第一阶段 `docs/PRD.md`（v0.1.0）之上。
> v0.1.0 已交付「能下载」；v0.2.0 解决「下得放心、找得到、看得见」——把下载闭环做扎实、把信息架构理顺。
> 状态/里程碑见 `ROADMAP.md`；逐里程碑实现计划见 `docs/plans/`。

日期：2026-06-08

---

## 1. 一句话定义

> v0.1.0 让用户**能**把文章下载下来；v0.2.0 让用户**下完即可确认、即可阅读，并能找回来**。

这一迭代不堆新功能，而是补齐第一版暴露出的体验与保真缺口。

## 3. 目标与非目标

### 3.1 目标

- **理顺信息架构**：导航与命名贴合「下载文章」这件事的本质，不让用户在"批量""书架"这类割裂概念间跳。
- **打通下载后的闭环**：下载完成即可在原地确认产出、立即阅读，无需切菜单再查找。
- **可追溯**：提供下载历史，"我刚下的那批"随时找得回、看得到、能重试。
- **流程可回退**：公众号下载的搜索 → 配置 → 运行各步可逆，不丢状态。
- **补保真与反馈**：修 PDF 跨页切图；让频控退避可见、让失败反馈引导下一步。
- **文库可管理**：从"平铺列表"进化到可排序、分组、批量操作的藏馆。

### 3.2 非目标（v0.2.0 明确不做）

- ❌ **其他形式内容保真**（公式 / 音视频卡片 / 文章合集 / 话题标签 / 小程序卡片 / 投票 / 留言）——需要一轮系统保真审计，单独立项，本迭代搁置。
- ❌ **mac 代码签名 / 公证、应用内自动更新**——未到推广阶段，列为后续。
- ❌ 多公众号 session 并存——经评估非真实痛点，不做。
- ❌ 云同步、账号体系、付费门槛（延续 v0.1.0 非目标）。

## 4. 信息架构变更（本迭代的骨架变化）

### 4.1 导航：四项 → 三项

| v0.1.0 | v0.2.0 | 说明 |
|---|---|---|
| 下载 | **下载** | 收纳两种下载模式（见 4.2） |
| 批量 | —（并入「下载」） | 批量本质也是下载，不该是独立顶级菜单 |
| 书架 | **文库** | 下载的是文章不是书，"书架"名不副实，统一为"文库" |
| 设置 | **设置** | 不变 |

**为什么**：用户的心智是"我要下载文章"，URL 和公众号只是两种入口，不是两个功能。把"批量"提到顶级导航，等于让用户先决定"我要用哪个功能"而不是"我要做什么"。

**对用户的影响**：导航更短、概念更少；下载相关的一切都在一个地方。

### 4.2 「下载」页内：两种模式

「下载」页顶部用显式开关切换两种下载方式，二者复用同一套格式选择、进度与结果区：

- **按 URL 下载**（**默认进入**）：粘贴一个/多个文章链接（即原「下载」页）。
- **按公众号下载**：搜索公众号 → 选号 → 选范围 → 下载（即原「批量」页，需登录）。

进入「下载」页默认停在「按 URL 下载」；开关常驻可见，一键切到「按公众号下载」。

### 4.3 命名统一

全局把「书架」改为「文库」：导航、空状态文案、阅读器入口、相关提示、（如有）文档与截图。English 标识符（路由 `/library`、`data-testid`）可保留，仅改面向用户的中文。

## 5. 功能需求

### R1 · 下载后闭环：就地确认与阅读

**问题**：下载完成后，用户无法在原地确认"是否按我要的格式下全了"，也无法直接阅读，必须切到文库再翻找。

**需求**：

- **结果区每条产物就地可操作**：
  - **立即阅读** —— 一键打开内置阅读器读这篇（md/html）。
  - **在文件夹中显示** —— 跳到磁盘位置。
  - **格式核对** —— 显式展示该篇**实际产出的格式集**（封面/md/html/pdf/meta 的徽章），与本次所选格式一眼对照，确认无缺漏。
- 失败/跳过项保留可见，失败项可单篇重试（延续 v0.1.0 行为）。

**对用户的影响**：下完即看、即核，不再"下载是个黑盒、还得去别处验货"。

### R2 · 下载历史

**问题**：当前下载结果只活在本次会话；关掉或切页就没了，"我昨天下的那批"无从找回。

**需求**：

- 「下载」页保留**最近下载历史**，跨会话持久化。
- 每条历史记录：时间、来源（URL 批 / 某公众号）、篇数与成功/跳过/失败统计、所选格式。
- 历史项可**展开看明细**、**再次阅读**其中文章、**重试失败项**。
- **保留策略**：默认保留 **1 年**，保留期可在设置中配置；支持**手动清空**。
- **关键语义（不可误解）**：历史只是"下载**动作**的记录"。过期清理与手动清空**只删动作记录，绝不删除已下载的文件，也不动文库条目**。下载下来的东西永远只由用户在文库里主动删除。
- **与文库删除的联动**：若某文章已被从文库删除，历史中对应项**标记为「已删除」**（保留记录、不隐藏），点击阅读时给出"原文已从文库删除"的提示。
- 与文库的边界（见 §6）：历史是"操作记录"，指向库内文章，不复制内容。

**对用户的影响**：下载有了时间线，可追溯、可复看；且"清历史"绝不会误伤已下载的文件。

### R3 · 公众号下载流程可回退

**问题**：按公众号下载时，点「开始下载」后配置区被进度区取代，无法回退查看/修改"搜号、范围、格式"，状态丢失。

**需求**：

- 流程分步：**搜索 → 选号 → 配置（范围+格式）→ 运行 → 结果**，每一步可返回上一步重配，已填状态不丢。
- 运行中可**取消并回到配置**（延续取消语义：停后续、保留已下）。
- 完成后可**改条件再下一批**，不必从搜号重来。

**对用户的影响**：配错了能改，不用推倒重来。

### R4 · PDF 保真：图片不跨页切断

**问题**：`export-pdf.ts` 裸 `printToPDF({ pageSize: 'A4' })`，无防跨页规则，长文里图片在 A4 页边界被拦腰切断。

**需求**：导出 PDF 前为页面注入打印 CSS，令图片/图表/代码块/表格在分页时**整体不被切断**（`break-inside: avoid`）。

**对用户的影响**：PDF 里的图是完整的，不再半张在上一页半张在下一页。

### R5 · 反馈引导：频控可见 + 失败有指引

**问题**：① 公众号批量命中频控时，core 在静默退避，GUI 仍显示"下载中"，看起来像卡死。② 失败只把原始报错（含英文超时文案 `fetch timeout after 20000ms: ...`）直接抛给用户，不引导下一步。

**需求**：

- **频控可见**：进度新增"退避中"态——明确告知"命中频控，N 秒后自动重试"，让用户知道是在等而非死了。
- **失败引导**：把原始错误归一为面向用户的话术 + 下一步建议（如"网络异常，请检查后重试""登录已过期，请重新登录"），原始信息可折叠备查。

**对用户的影响**：系统状态透明，出问题知道该干什么。

### R6 · 文库组织能力

**问题**：文库只有"按标题搜索 + 删除"，攒到几十上百篇后无法组织。

**需求**（可分批交付）：

- **排序**：按下载时间 / 发布时间。
- **按公众号分组或筛选**。
- **批量选择 + 批量删除**。

> 注：整本导出（合并 PDF/EPUB）本迭代不做（见 §10）。需要打包带走时，用"批量选中 → 在文件夹中显示"兜底。

**对用户的影响**：文库从"垃圾堆"变成"藏馆"。

### R7 · 改库根目录的提示完善

**问题**：设置里改文章库根目录，仅提示"旧文章不会自动迁移"，没说清后果。

**需求**：提示补全——明确"改后文库列表会暂时变空，旧文章仍在原目录、可改回找回"，避免用户误以为数据丢了。

**对用户的影响**：知情，不慌。

## 6. 关键产品概念：文库 vs 下载历史（边界定义）

这是本迭代最需要厘清的概念，决定存储与 UI 设计：

| | **文库（Library）** | **下载历史（Download History）** |
|---|---|---|
| 是什么 | 你拥有的**全部文章**的藏馆 | 你做过的**下载动作**的时间线 |
| 组织方式 | 按文章（公众号/时间分组、搜索） | 按批次/时间 |
| 用途 | 长期管理：找、读、分组、删 | 追溯：刚下那批怎么样、复看、重试 |
| 入口 | 顶级导航「文库」 | 「下载」页内的"最近下载"区 |
| 存储 | `library.json`（全量索引，已有） | 轻量历史索引，**指向**库内文章，不复制内容 |
| 阅读器 | 从文库点进 | 从历史/结果点进（同一个阅读器） |

> 一句话：**文库回答"我有什么"，历史回答"我刚做了什么"。** 二者共用文章与阅读器，但不是同一个列表。

## 7. 数据与存储影响

- **下载历史索引**：新增一份轻量持久化（如 `history.json`），每条记录一次下载操作（时间、来源、格式、各文章 id 与结果）。指向 `library.json` 中的文章，不冗余正文。保留期默认 1 年、可配置；过期清理与手动清空**只增删 `history.json` 自身记录，不触碰文章文件与 `library.json`**。
- **设置新增项**：下载历史保留期（默认 1 年）、清空下载历史的入口。
- **library.json**：维持现有结构；R6 的排序/分组基于其已有字段（发布时间 v0.1.0 已修复可用），必要时补字段。
- 不引入数据库（延续 v0.1.0 约束：文件系统 + JSON 索引）。

## 8. 验收标准

- 导航为「下载 / 文库 / 设置」三项；「下载」页可在"按 URL / 按公众号"间切换；全局无"书架""批量"字样。
- 一次 URL 下载完成后，**无需切换菜单**即可：核对每篇实际产出格式、就地打开阅读、在文件夹显示。
- 关闭并重开应用后，"最近下载"仍在，可复看与重试。
- 按公众号下载：开始后可回退改范围/格式/换号且状态不丢；取消后回到配置。
- 含图长文导出的 PDF：图片无跨页截断。
- 批量命中频控时，界面明确显示"退避中/N 秒后重试"；任一失败的提示包含可执行的下一步。
- 文库可按时间排序、按公众号筛选、批量删除。
- 改库根目录的提示说清"列表暂空、旧文在原目录"。
- 既有单测 / e2e 全绿（导航与命名变更需同步更新 e2e 选择器）。

## 10. 已决策（2026-06-08 安哥拍板）

- **下载历史保留**：默认保留 **1 年**，可配置，支持手动清空。清空/过期**只清动作记录，绝不删除已下载文件或文库条目**。（落入 R2 / §7）
- **「下载」页双模式默认**：默认进入**按 URL 下载**，页面常驻开关可切换。（落入 §4.2）
- **整本导出**：本迭代**不做**（与"文章不是书"的命名哲学有张力；批量选中+在文件夹显示可兜底）。从 R6 移除，连 stretch 也不列。
- **文库删除联动**：从文库删文章后，历史中对应项**标记为「已删除」**，不隐藏。（落入 R2）
